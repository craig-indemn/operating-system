# System Landscape Research: CLI & MCP Server Foundation

## Executive Summary

The Indemn platform has **two API backends** (Copilot Server for V1 agents, Percy Service for V2 agents), **comprehensive architectural documentation**, and a **complete OpenAPI spec** for Copilot Server. However, there are **zero existing SDKs, CLI tools, or MCP servers**. All inter-service and frontend-to-backend communication is via direct REST calls with no abstraction layer.

---

## Two API Layers

### Copilot Server (V1 Agents — Production)
- **Tech**: Node.js Express + WebSocket
- **Port**: 3000
- **Route files**: 65
- **Auth**: JWT (dynamic secret per resource type) + Firebase + Basic + Google OAuth
- **Database**: MongoDB (tiledesk database)
- **OpenAPI spec**: Yes (64KB, copilot-server/docs/openapi.yaml)

**Key API surfaces:**
| Domain | Endpoints | What They Do |
|--------|-----------|-------------|
| AI Agents | `/ai-studio/bots` | Full CRUD for V1 agents |
| Functions/Tools | `/ai-studio/bots/:id/functions` | CRUD for bot functions, parameter management, API testing |
| Knowledge Bases | `/knowledge-bases` | CRUD for KBs, data sources, import/export |
| Bot Config | `/ai-studio/bots/:id/configurations` | System prompts, LLM selection, greeting, KB mapping |
| Conversations | `/requests`, `/conversation` | Conversation management |
| Webhooks | `/subscriptions` | Webhook registration and management |
| Voice | `/voices`, `/voiceConfigurations` | Voice agent configuration |
| LLM Models | `/llmModels` | Available model listing |
| Auth | `/auth` | Signup, signin, password reset |
| Organization | `/organization` | Org management |

**All endpoints follow**: `{BASE}/organization/{orgId}/{resource}`

### Percy Service (V2 Agents — Platform V2)
- **Tech**: Python FastAPI + WebSocket
- **Port**: 8003
- **Auth**: Optional JWT with org isolation (controlled by AUTH_ENABLED env var)
- **Database**: MongoDB (indemn_platform database)
- **OpenAPI spec**: No (auto-generated by FastAPI at /docs)

**Key API surfaces:**
| Domain | Endpoints | What They Do |
|--------|-----------|-------------|
| V2 Agents | `/api/v1/agents` | CRUD for config-driven agents |
| Templates | `/api/v1/templates` | Graph template management |
| Components | `/api/v1/components` | Component registry |
| Connectors | `/api/v1/connectors` | External integration management |
| Evaluations | `/api/v1/scenarios`, `/simulations` | Evaluation orchestration |
| Jarvis | `/api/v1/jarvis` | Bulk job operations |
| WebSocket | `/ws` | Real-time Jarvis chat |

## Authentication Models

### Current State
| System | Method | Details |
|--------|--------|---------|
| Copilot Dashboard | Firebase Auth | Email/password + social (Google, Microsoft) + MFA |
| Copilot Server | JWT + Basic + Firebase | Dynamic JWT secret per resource type |
| Percy Service | JWT (optional) | Simple JWT with org isolation, can be disabled |

### Key Auth Details
- **Copilot Server JWT**: Complex — secret is resolved dynamically per audience claim (bot, project, subscription each have their own secret)
- **Percy Service JWT**: Simple — single JWT_SECRET, extracts user_id + org_id, detects @indemn.ai employees
- **No API key system exists** — no way for external consumers to authenticate without Firebase
- **No OAuth2 client credentials flow** — no machine-to-machine auth

## Data Models (What the CLI/MCP Would Manage)

### Agent (faq_kb in Copilot Server)
- name, description, channels, language, webhook settings, lifecycle state
- Has: configurations (prompts), functions (tools), KB mappings, LLM config, voice config

### Bot Configuration
- system_prompt, LLM model, greeting message, quick replies, KB namespace/index

### Functions/Tools (botTools)
- Types: rest_api, human_handoff, custom_tool, transfer_call, end_call, faqs-retriever
- Has: parameters with types, auth configuration (bearer, api-key, basic)

### Knowledge Bases
- Types: QNA, URL, FILE
- Has: data sources (QA pairs, URLs, files), import/export

### Evaluations (Percy Service)
- Rubrics (3-6 universal rules), test sets (scenarios + single-turn items)
- Simulation runs with pass/fail per criteria
- Directive allocation (universal vs workflow-specific)

## Existing Skills (Percy Service)
Three evaluation workflow skills already exist:
1. **eval-orchestration** — Full evaluation lifecycle with 6 workflow types
2. **test-set-creation** — Multi-turn scenarios, voice simulations, single-turn diagnostics
3. **rubric-creation** — Universal behavioral rubrics with mandatory baselines

These skills call evaluation service APIs via HTTP connectors. They are NOT CLIs or MCPs — they're Claude Code skill files that guide API usage.
