Event-Driven Architecture at Scale — \"Every Change Generates Events\"\n57\t\n58\t### The Practical Reality\n59\t\n60\tThe design says \"every entity change generates messages.\" This is the Event Notification pattern. At scale, this creates known challenges:\n61\t\n62\t**Event Storms** — A single operation can cascade. Binding a policy: Quote transitions -> message -> Policy created -> message -> Payment initiated -> message -> Commission calculated -> message -> Certificate generated -> message -> Notification sent. One user action = 6+ messages. If each message triggers further entity changes, the graph can explode.\n63\t\n64\t**Mitigation strategies that work:**\n65\t1. **Distinguish internal vs. external events.** Entity state changes within a single service operation are internal. Only the final result publishes external messages. The `BindingService.bind()` method changes Quote, creates Policy, etc. internally. One message out: \"binding.completed\" with all the context.\n66\t2. **Batch/coalesce.** Multiple field changes on the same entity in the same operation = one message, not one per field. Debounce rapid changes.\n67\t3. **Priority tiers.** Not all messages are equal. State transitions > field updates > relationship changes. Route high-priority messages first.\n68\t4. **Load shedding.** Under extreme load, drop low-priority messages (field change notifications) while preserving critical ones (state transitions).\n69\t\n70\t**Backpressure** — When message production exceeds consumption:\n71\t- **Queue depth monitoring** — alert when any actor's queue grows beyond threshold\n72\t- **Rate limiting at the source** — slow down message generation if queues are deep\n73\t- **Consumer groups / horizontal scaling** — multiple workers per role for high-volume message types\n74\t- **Dead letter handling** — messages that can't be processed after N retries go to a dead letter queue for manual review\n75\t\n76\t**Filtering** — Not every actor needs every message. The routing rules (defined by Role) filter messages before they hit actor queues. This is the most important performance mechanism. Without it, every actor gets every message and must discard irrelevant ones.\n77\t\n78\t### The Right Granularity\n79\t\n80\t**Don't emit on every field change.** Emit on:\n81\t- Entity creation\n82\t- Entity deletion\n83\t- State machine transitions (the most important)\n84\t- Explicit business operations (@exposed methods)\n85\t- Relationship changes (entity linked/unlinked)\n86\t\n87\t**Don't emit on:**\n88\t- Draft saves / partial updates (until explicitly \"submitted\")\n89\t- Internal computed field updates\n90\t- Read operations\n91\t- Timestamp updates\n92\t\n93\tThis keeps the message volume manageable. A busy insurance agency might process 100 submissions/day. Even with 10 messages per submission lifecycle, that's 1,000 messages/day — trivial for any messaging system. Scale concern is future-state with hundreds of organizations.\n94\t\n95\t### The Outbox Pattern — Critical for MongoDB\n96\t\n97\tThe biggest risk in \"every change generates events\": the entity is saved to MongoDB but the message fails to publish (or vice versa). This creates inconsistency.\n98\t\n99\t**The Transactional Outbox Pattern solves this:**\n100\t1. When an entity changes, write the message to an `outbox` collection in the SAME MongoDB transaction as the entity save\n101\t2. When a document changes, the change stream emits an event. No outbox needed. But: change streams only tell you WHAT changed, not WHY. A state transition and a field update look the same. You lose business intent. Context Enrichment on Messages\n379\t\n380\t### The Problem\n381\t\n382\tWhen a message says \"Submission SUB-001 transitioned to triaged,\" the actor receiving it needs context: What's in the submission? The state machine does not account for repeated extensions. You need a binder that can loop back on itself: active -> extended -> active -> extended -> active. This is not a pathological case. It is routine in commercial lines.\n\n**Multi-location / multi-state policies**\n\nA single BOP or property policy can cover 47 locations across 12 states. Each location has its own rated premium, its own coverages, its own limits. Some states require surplus lines filing. Some require specific endorsements. The entity model has Coverage as an aggregate child of Policy, but there is no Location entity. Location-level rating, location-level endorsements, and location-level compliance requirements are the daily reality of commercial lines. A \"flexible data\" dict is not sufficient here. Every carrier rates and endorses at the location level. This is structural, not configurable.\n\n**Surplus lines complexity**\n\nSurplusLinesTransaction is modeled as a standalone entity with a lifecycle of drafted -> filed -> accepted/rejected -> tax paid. The reality is that surplus lines compliance is per-state, per-line, per-policy, and involves a diligent search that must be documented before placement. The tax rates are different by state. Some states require the tax to be collected from the insured. Some require it to be paid by the broker. Some states (like Florida) have additional stamping fees. Some require filing within 30 days, some within 60. The entity model treats this as one thing. In practice, a single policy placement in 3 states requires 3 separate surplus lines transactions, 3 different tax calculations, 3 different filing deadlines, and the diligent search must cover licensed carriers in all 3 states. This is per-state regulatory machinery that does not fit in a single entity with a simple state machine.\n\n**Split placements**\n\nSome large risks are split across multiple carriers. 60% with Carrier A, 40% with Carrier B. Same policy, same insured, same coverage terms, different carriers bearing different shares. This creates parallel premium flows, parallel commission flows, parallel bordereaux reporting, and parallel claims handling. The entity model has Policy with a single Carrier relationship. A split placement breaks this assumption entirely. You need either a Placement entity that sits above Policy, or you need multiple Policy records that are linked as one conceptual placement. Neither pattern exists in the model.\n\n**Agency bill vs. direct bill**\n\nThe billing method fundamentally changes the money flow and the commission flow. Agency bill means the agency collects premium from the insured, takes their commission off the top, and remits the net premium to the carrier. Direct bill means the carrier bills the insured directly and pays the agency a commission separately. The entity model has Payment, Invoice, and CommissionTransaction but does not distinguish these two radically different billing flows. The trust account obligations, the timing of commission recognition, and the accounting treatment are completely different. This is not a configuration option on an existing entity. It is a different financial architecture.\n\n**Cancellation-reinstatement cycles**\n\nThe Policy state machine has cancelled as a terminal state. In reality, policies get cancelled for non-payment, the insured pays, and the policy gets reinstated. Sometimes with a lapse, sometimes without. Sometimes the reinstatement requires new underwriting. Sometimes it is automatic. The model has no reinstatement path from cancelled. PolicyTransaction is supposed to absorb cancellation, but reinstatement as a PolicyTransaction type that reverses a cancellation is a conceptually different operation than an endorsement or renewal.\n\n**Certificate holder endorsements**\n\nCertificates are modeled as a lightweight entity. In practice, certificate requests drive enormous operational volume. A contractor might need 50 certificates a month, each with different additional insured requirements, each requiring a specific endorsement on the underlying policy. The connection between a certificate request and an endorsement on the policy is the real workflow. The model treats certificates and policies as separate entities with a reference relationship, but the actual work is: certificate holder requires additional insured status -> endorsement must be issued on the policy -> certificate can only be issued after endorsement is confirmed. This is a tightly coupled cross-entity workflow that the message routing model will struggle to coordinate reliably.\n\n## 2. What Is Missing\n\n**Activity/Journal/Diary system.** Every AMS has an activity log. Not an audit trail of entity changes, but a conscious diary system where people note \"Called John at Hartford about the Smith submission, he said underwriting needs loss runs going back 5 years, will follow up Thursday.\" The system tracks these notes, the follow-up dates, and reminds people. This is the nervous system of an insurance office. Every person in the office lives in their diary. The domain model has Task and Interaction, but neither is a diary system.\n\n**Suspense/follow-up management.** Related to the diary but distinct. Suspense items are dated reminders attached to specific transactions. \"Policy expires 4/15, renewal quote needed by 3/15, marketing started 2/15.\" These are not tasks. They are temporal checkpoints in a process. The model has no concept of a suspense system.\n\n**Download/IVANS integration.** This is how policies actually enter an agency management system. Carriers send ACORD download transactions (AL3, IVANS) that automatically update policy data in the AMS. New business, endorsements, cancellations, renewals, commission statements. The agency does not manually enter most of this data. It flows in electronically. The entity model has no concept of inbound carrier data feeds. The integration layer mentions \"adapter pattern on entity operations\" but download processing is a massive, standards-based, carrier-specific data integration that deserves its own design, not a hand-wave to adapters.\n\n**Accounting / general ledger integration.** Insurance agencies are regulated financial entities. Premium trust accounting, operating account management, bank reconciliation, carrier payables, commission receivables, producer payables. The financial entities in the model (Payment, Invoice, Commission) are transaction records, but there is no chart of accounts, no journal entries, no trial balance, no financial reporting structure. An agency cannot run on this without an external accounting system, which means the \"operating system\" claim is immediately false for any agency that currently uses their AMS for accounting.\n\n**ACORD forms.** The insurance industry runs on ACORD forms. ACORD 125 (commercial insurance application), ACORD 25 (certificate of liability insurance), ACORD 27 (evidence of property insurance), ACORD 126 (commercial general liability section), ACORD 130 (workers compensation application). These are not templates. They are standardized industry forms with specific field mappings, specific layouts, and specific regulatory significance. The model has Template and Document but no understanding of ACORD form standards. This is table stakes for any system that claims to be an insurance operating system.\n\n**Remarketing / market comparison.** When you remarket an account, you send the same submission to 5-8 carriers and compare the quotes side by side. The model has multiple Quotes per Submission, but there is no structured comparison entity, no way to track which carriers were approached, which declined (and why), and which quoted. The Panel/Market concept was explicitly dropped from the model. In practice, \"who did we market this to and what happened with each carrier\" is fundamental to E&O protection and client service.\n\n**E&O (Errors and Omissions) documentation.** Every coverage recommendation, every declination, every conversation about coverage options must be documented for E&O protection. \"We offered umbrella and the insured declined\" is a liability shield. The entity model has no concept of a coverage recommendation or declination record. This is not a nice-to-have. It is a legal necessity that every experienced insurance professional thinks about every day.\n\n**Rate comparison / indication tracking.** Before a formal quote, agencies often get indications from carriers. These are informal, non-binding premium estimates. They inform the marketing strategy. An indication is not a Quote. It is a pre-quote signal that guides where to submit. The model jumps from Submission to Quote with nothing in between.\n\n## 3. Where Configurable Routing Fails\n\n**Same workflow, different legal obligations.** A managing general agent and a retail agency both process submissions. But the MGA has binding authority and the retail agency does not. This is not a routing difference. It is a fundamentally different legal and operational reality. The MGA can bind coverage and issue the policy. The retail agent can only submit to the carrier and wait. The \"submission processing\" workflow is structurally different, not just routed differently. The entity state machine for Submission needs different states and different transitions depending on whether the organization has binding authority.\n\n**State-specific regulatory requirements.** California requires a specific notice for earthquake coverage declination. Florida has specific windstorm mitigation credits. Texas has unique surplus lines filing rules. New York has specific disclosure requirements. These are not routing rules. They are conditional logic that changes which fields are required, which documents must be generated, which filings must be made, and which timelines apply. Per-org routing does not capture per-state regulatory variation that applies regardless of the organization.\n\n**Carrier-specific submission requirements.** Hartford wants a 5-year loss run. Travelers wants a 3-year loss run plus a photo of the building. Cincinnati wants a signed supplemental application. These are not routing differences. They are data requirements that change what the submission must contain, what documents must be attached, and what validation rules apply. The Appetite entity captures what a carrier will write, but the specific data requirements per carrier per product are operational details that change how the submission processing workflow works, not just where it routes.\n\n**Compensation structures.** Some agencies pay producers on collected premium. Some pay on written premium. Some have sliding scale commissions that change based on loss ratio. Some have contingency agreements that pay bonuses annually based on profitability. Some have override agreements where the agency gets an extra percentage on certain lines. The CommissionSchedule entity captures rates, but the calculation logic is organization-specific, carrier-specific, and often product-specific. This is not configurable routing. It is configurable business logic that differs per organization at the calculation level, not the routing level.\n\n## 4. The People Problem\n\n**Producer codes are the real identity.** In the carrier's system, a producer code identifies who sold the policy. This determines who gets paid. But a single person can have multiple producer codes with the same carrier (personal lines vs. commercial lines). An agency can have multiple producer codes with the same carrier (different branch offices). A sub-producer can have their own code that rolls up to the agency's code. The Party/Role model with RetailAgent as a role on Contact does not capture the producer code as a first-class concept. Producer codes are the joints of the distribution system. They determine money flow, reporting, and carrier relationships.\n\n**Hierarchical distribution.** Carrier -> MGA -> Wholesale Broker -> Retail Agent -> Sub-Producer. Each level in this chain has its own agreement, its own compensation, its own authority, and its own reporting obligations. The model has Carrier, Agency, RetailAgent, and DistributionPartner. But the hierarchical nature of insurance distribution, where there can be 3-4 intermediaries between the carrier and the insured, is not modeled. An MGA that uses sub-agents needs the system to understand the full chain.\n\n**Appointments are per-state, per-line, per-entity type.** ProducerAppointment is modeled as per-producer, per-carrier, per-state. In reality, appointments can also be per-line-of-business. A producer might be appointed to sell Hartford's commercial auto in New York but not their workers' compensation in New York. The granularity matters for compliance. Selling a product you are not appointed to sell is a regulatory violation.\n\n**Agency networks and clusters.** Many independent agencies belong to networks (SIAA, ISU, Keystone) or clusters that aggregate premium for better carrier access and compensation. The network relationship changes which carrier appointments are available, what commission rates apply, and what reporting is required. This is another layer of distribution hierarchy that the model does not capture.\n\n## 5. What Would Make Me Say \"This Doesn't Understand My Business\"\n\n**The state machines are too simple.** The Policy lifecycle of issued -> active -> endorsed -> renewed -> cancelled/expired does not capture reality. A policy can be: active and pending cancellation (notice sent but not yet effective), cancelled flat (never should have been issued), cancelled pro-rata (mid-term), cancelled short-rate (penalty), non-renewed (carrier chose not to renew), conditionally renewed (renewed with restrictions), audited (final premium adjustment on a policy that has already expired). Each of these has different financial implications, different regulatory requirements, and different operational workflows. A generic state machine with 6 states is what someone designs when they have read about insurance but not operated in it.\n\n**No concept of \"the file.\"** Insurance people think in terms of files. The client file. The policy file. The claim file. A file is not an entity. It is an aggregation point. Everything related to a client or a policy or a claim, in one place, in chronological order. Activities, documents, emails, notes, endorsements, payments, certificates, correspondence. The model has entities with relationships, but it does not have the concept of the file as the organizing principle. When an account manager opens a client's file, they expect to see everything, in order, across all entities. The entity model as designed would require them to navigate between Insured, Policy, Submission, Certificate, Claim, Payment, Interaction, Correspondence, and Document as separate views. That is not how insurance people work.\n\n**No concept of book of business.** A producer's book of business is the fundamental unit of insurance value. It is the collection of policies that a producer controls. Books get bought and sold. Books get transferred between producers. Books determine a producer's compensation, their value to the agency, and their leverage with carriers. The model has no way to express \"all policies where RetailAgent X is the servicing agent\" as a first-class concept with its own metrics, its own value, and its own lifecycle.\n\n**No concept of appetite vs. risk tolerance.** The Carrier entity has Appetite as an aggregate child, which captures \"what the carrier will write.\" But there is a difference between appetite (what they want) and tolerance (what they will accept under the right conditions). Every carrier has written guidelines, and then there is what the underwriter will actually do when you call them. The distinction between the published appetite and the actual underwriting flexibility is critical for producers who know their carriers. A system that only captures the published appetite misses half the value of an experienced producer's carrier relationships.\n\n**The auto-generated UI assumption is dangerous.** Insurance operations require dense, information-rich interfaces. An account manager looking at a policy needs to see the declaration page summary, all endorsements, premium by coverage, payment status, certificate holders, upcoming expirations, open claims, recent activity, and related policies on the same account, all on one screen. Auto-generated UIs from entity definitions produce CRUD forms, not operational dashboards. The claim that \"the UI auto-generates from the entity definitions and role permissions\" will produce a system that looks like a database admin tool, not an insurance management system. Every insurance professional who has used Applied Epic, AMS360, or even a well-configured Salesforce instance will immediately see the difference.\n\n**No concept of time-sensitivity and regulatory deadlines.** Insurance is a business of deadlines. Cancellation notices must be sent X days before effective date (varies by state and reason). Surplus lines taxes must be filed within Y days. Renewal offers must be made Z days before expiration. Claims must be acknowledged within W days. The model has entities with lifecycles but no built-in concept of regulatory deadlines driving the work. The message escalation system (if unprocessed, re-route after timeout) is a generic mechanism. Insurance needs deadline-aware entities that know their own regulatory calendar.\n\n---\n\n**Bottom line:** The two-primitive model (Entity + Message) is a sound architectural pattern. It could work as an insurance platform. But the domain model as currently designed reflects someone who has studied insurance domain modeling, not someone who has run an insurance office. The entities are right at the 30,000-foot level and wrong at the operational level. The state machines are too simple. The financial model is too shallow. The regulatory awareness is absent.
