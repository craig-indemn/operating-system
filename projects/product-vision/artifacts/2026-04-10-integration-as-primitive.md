# Integration as the Sixth Primitive

## Decision

**Integration is a bootstrap kernel primitive, alongside Organization, Actor, and Role.** It is not just one of many domain entities — it is the infrastructure primitive for connecting the OS to the outside world. Elevating it makes explicit what the architecture already assumed but never formally stated: every entity that does anything external depends on it, and it has its own lifecycle, credential management, and access model that the kernel must understand.

## The Updated Primitive Set

Six bootstrap primitives the kernel knows specially:

| # | Primitive | What It Is |
|---|-----------|-----------|
| 1 | **Entity** | Domain data with structure, lifecycle, CLI, API, auto-generated skills |
| 2 | **Message** | Work items in the queue — generated by entity changes, routed by watches |
| 3 | **Actor** | Identity (human or automated) that processes triggers by executing CLI commands |
| 4 | **Role** | Permissions (what you can access) + watches (what comes to you) |
| 5 | **Organization** | Multi-tenancy scope — environments are orgs |
| 6 | **Integration** | External system connection — provider, credentials, ownership, adapter dispatch |

All six are bootstrap entities: managed through the same CLI/API as domain entities, but the kernel depends on them for core behavior (tenancy scoping, identity, routing, external connectivity). You cannot build a system on the OS without understanding these six. Everything else is domain.

## Why Integration Is a Primitive

### It's infrastructure, not domain

Integration is not an insurance concept. Every system built on the OS needs external connectivity: email providers, messaging platforms, payment processors, CRMs, AMSs, carrier APIs, AI services, storage buckets. The OS cannot do anything beyond its own database without Integration. This is the same scope of universality as Actor (every system needs participants) or Organization (every system needs tenancy).

### It has its own lifecycle

An Integration moves through states: `configured → connected → active → error → reconfigured`. These are infrastructure transitions, distinct from the business lifecycles of domain entities. Connection testing, credential rotation, provider versioning, and health monitoring all apply to Integration specifically, not to the entities it serves.

### It owns credential management

Every external connection has secrets. These must be stored securely (not in the entity document), rotated safely, audited for access, and scoped correctly (who can use this credential?). Making Integration a primitive gives the kernel one place to handle this uniformly across all providers and all entity types.

### It's shared across entity types

One Outlook Integration serves Email, Calendar, Contacts, and potentially Tasks. One Stripe Integration serves Payment, Invoice, and Subscription. If Integration were just a regular domain entity, this sharing would be ad-hoc. As a primitive, the kernel can dispatch any entity's external operation to the right Integration record automatically.

### It has both org-level and actor-level ownership

Org-level integrations are shared by role (the company's Outlook, the carrier API). Actor-level integrations belong to a specific actor (JC's personal Gmail, the associate's own Slack token for sending follow-ups). Both are Integration records — the primitive handles both cases through an `owner` field. This wasn't addressed in the prior architecture and would have required a new concept if Integration stayed at the domain level.

## Integration Is the Abstraction. Adapter Is the Implementation.

These are not competing concepts — they are two halves of one system at different levels:

| Layer | What | Where | Example |
|-------|------|-------|---------|
| **Integration** (primitive) | The record: provider, owner, credentials, status | MongoDB (bootstrap entity, managed via CLI) | "GIC's Outlook connection, owned by org, active" |
| **Adapter** (implementation) | The kernel code that executes provider-specific operations | Python in the OS codebase (platform code) | `OutlookEmailAdapter.fetch()`, `StripePaymentAdapter.charge()` |

When an entity operation needs external connectivity:

```
indemn email fetch-new
  → kernel looks up Integration for this org (and/or actor)
  → reads Integration.provider → "outlook"
  → finds adapter: ADAPTER_REGISTRY["outlook"] → OutlookEmailAdapter
  → reads Integration.secret_ref → fetches credentials from AWS Secrets Manager
  → instantiates adapter with credentials
  → invokes adapter.fetch() → returns Email entity data
  → entity framework creates Email entities, kernel takes over
```

The Integration entity knows WHICH provider and WHICH credentials. The Adapter class knows HOW to talk to that provider. Neither can work without the other.

### What this reconciles

The earlier session-4 framing ("an Outlook email IS an Email entity, integration layer collapses into entity framework") was correct in intent but left the Integration concept underspecified. It sounded like Integration disappeared, but it didn't — it's still needed to hold the config and credentials. The collapse was about **removing a separate "integration layer" as an architectural concept**, not about removing the Integration record itself. Adapters are no longer a separate "layer 4" — they're kernel code that Integration dispatches to.

Elevating Integration to a primitive makes this clean: there is no integration layer, but there is an Integration primitive, and it uses adapters (kernel code) as its implementation mechanism.

## The Integration Record

```python
class Integration(Entity):  # bootstrap entity — kernel knows about it specially
    # Identity and ownership
    name: str                        # "GIC Outlook", "JC Personal Gmail"
    owner_type: Literal["org", "actor"]
    owner_id: ObjectId               # org_id or actor_id

    # Provider and configuration
    system_type: str                 # "email", "calendar", "messaging", "payment", "ams", "carrier"
    provider: str                    # "outlook", "gmail", "slack", "stripe", "applied-epic", "usli"
    provider_version: str            # "outlook_v2", "stripe_2024-06-20" (for adapter versioning)
    config: dict                     # Provider-specific non-secret config

    # Credentials (never stored inline)
    secret_ref: str                  # Path in AWS Secrets Manager

    # Access control
    access: dict                     # For org-level: {"roles": ["...", "..."]}
                                     # For actor-level: null (only owner can use)

    # Lifecycle
    status: Literal["configured", "connected", "active", "error", "paused"]
    last_checked_at: Optional[datetime]
    last_error: Optional[str]

    # Standard kernel fields
    org_id: ObjectId                 # even actor-level integrations are scoped to an org
    created_at: datetime
    created_by: ObjectId             # the actor who set it up
    version: int                     # optimistic concurrency
```

## Credential Resolution

When any entity operation needs external connectivity, the kernel resolves the Integration to use. The resolution order encodes the access model:

```
Actor invokes `indemn email fetch-new`:

1. Does the calling actor have a personal Integration
   where system_type="email" and owner_id=actor.id?
     → use it.

2. Does the org have an Integration where system_type="email"
   and access.roles contains any of the actor's roles?
     → use it.

3. No match → fail with clear error:
   "No email integration available for this actor.
    Add one with `indemn integration create` or request
    access to an org-level integration from an admin."
```

Actor-level takes precedence over org-level. This means an individual actor can override the default with their own credentials (an account manager syncs their own Gmail instead of using the company's shared Outlook), and the system naturally does the right thing. If no personal integration exists, the org-level fallback kicks in based on role permissions.

For operations that must be org-level regardless (paying a carrier, filing a regulatory submission), the adapter or the capability can specify "org-only" and the resolver skips step 1.

## Credential Management

### Storage

Credentials never live in MongoDB. The Integration record stores `secret_ref`, a path in AWS Secrets Manager:

```
/indemn/{env}/org/{org_id}/integration/{integration_id}
/indemn/{env}/actor/{actor_id}/integration/{integration_id}
```

The adapter fetches credentials at runtime via the secret_ref. No code path reads credentials from MongoDB. No API response or CLI output contains credentials.

### Rotation

Rotation is a first-class CLI operation:

```bash
indemn integration rotate-credentials INT-001
# Triggers provider-specific rotation (OAuth refresh, API key regeneration)
# Updates Secrets Manager, updates secret_ref if path changes
# Records rotation in changes collection
# Alerts owner if rotation fails
```

Some providers require user interaction (re-consenting to OAuth scope). For those, rotation creates a Draft entity or similar work item in the owner's queue: "Your Outlook token expires in 7 days. Click to re-authorize."

### Audit

Every credential access is logged. The changes collection records:
- Integration creation and modification
- Credential rotation events
- Integration status changes
- Access denials (actor tried to use an integration they don't have permission for)

Combined with OTEL traces, every external operation can be traced from the entity method call through the Integration lookup, credential fetch, and adapter invocation.

## CLI Surface

```bash
# Create an org-level integration
indemn integration create \
  --owner org \
  --name "GIC Outlook" \
  --system-type email \
  --provider outlook \
  --access-roles "underwriter,operations,admin"

# Configure it (sets connection parameters and triggers credential setup)
indemn integration configure INT-001 --config @outlook-config.json
# This launches an OAuth flow for the owner, stores the result in Secrets Manager,
# and updates the Integration record with the secret_ref.

# Test the connection
indemn integration test INT-001

# Activate
indemn integration transition INT-001 --to active

# Create an actor-level integration (personal Gmail for an individual actor)
indemn integration create \
  --owner actor \
  --actor jc@gicunderwriters.com \
  --name "JC Personal Gmail" \
  --system-type email \
  --provider gmail

# Manage: rotate, pause, query
indemn integration rotate-credentials INT-003
indemn integration pause INT-001
indemn integration list --owner org --system-type email
indemn integration list --owner actor --actor jc@gicunderwriters.com
indemn integration history INT-001  # changes collection entries for this integration
```

## Adapter Versioning

The review finding about external API changes (Outlook Graph API has had 3 breaking changes in 18 months) fits here naturally:

- `provider_version` on the Integration record: `"outlook_v1"`, `"outlook_v2"`.
- Adapter registry keyed by version: `ADAPTER_REGISTRY["outlook_v2"] = OutlookV2Adapter`.
- Migration: `indemn integration upgrade INT-001 --to outlook_v2 --dry-run` validates the new adapter version against the existing Integration config, runs a test call, and either commits or reports compatibility issues.

This gives the platform team a path to ship provider adapter updates without forcing every org to reconfigure at the same time. Orgs upgrade on their own schedule.

## Relationship to Other Primitives

| Primitive | How Integration Relates |
|-----------|------------------------|
| **Organization** | Every Integration is scoped to an org (even actor-owned integrations — an actor belongs to an org) |
| **Actor** | Actors can own personal Integrations. An actor's effective Integration set is their personal ones plus the org-level ones their roles grant access to |
| **Role** | Org-level Integration access is role-based: `access.roles` specifies which roles can use an org-owned Integration |
| **Entity** | Any domain entity operation that touches the outside world dispatches through an Integration. The entity doesn't know which provider — that's Integration's job |
| **Message** | Integration changes (connect, error, rotate) generate messages via watches. Operations can watch for `Integration:status_changed[status=error]` and route to an ops role for investigation |

## What This Does NOT Change

- **The adapter pattern** remains exactly as designed: Python classes in the OS codebase, organized in an adapter registry, implementing per-provider operations.
- **Entity operations** like `indemn email fetch-new` work the same way from the caller's perspective.
- **AWS Secrets Manager** is still the credential store.
- **Existing decisions** about data architecture, the message queue, Temporal execution, watches, skills, and rules are all unchanged.

The only thing that changes is the explicit recognition that Integration is infrastructure, not domain. It gets a formal place in the primitive set and a clearer relationship to the code that implements it.

## What's Now Resolved

- **User credential management** (Craig's open question): actor-level Integration with `owner=actor` handles personal credentials. Same primitive as org-level, different owner.
- **Adapter versioning** (review finding): `provider_version` field on Integration, registry lookup by version, migration CLI.
- **The "integration layer" ambiguity**: there is no integration layer — there is an Integration primitive and its adapter implementations. Same architecture, clearer framing.
- **Credential rotation and audit**: first-class CLI operations, recorded in the changes collection, traceable via OTEL.
- **Integration health monitoring**: Integration has its own status field. Watches on Integration state changes surface problems to operations roles automatically.

## Open Questions

- **Adapter contribution path for Tier 3**: when a Tier 3 developer needs to connect to a provider the OS doesn't support yet, what's the path? Contributing a new adapter to the kernel (platform PR) is the obvious answer. Whether there's a managed-extension mechanism (custom adapter loaded at runtime) is an open design question. Deferred until we have a real Tier 3 use case.
- **Consent flows for OAuth providers**: many providers require interactive browser consent. The `indemn integration configure` CLI command needs a design for how it launches and captures browser consent in a headless context (for scripts, agents, and remote CLI use). Likely a short-lived web endpoint that the CLI opens for the user.
- **Integration dependencies**: does a domain entity declare which Integrations it depends on, or does it discover them at runtime? For now, runtime discovery via `system_type` lookup is sufficient. Explicit dependencies could be added if we need static analysis ("does this org have everything it needs to run this workflow?").
