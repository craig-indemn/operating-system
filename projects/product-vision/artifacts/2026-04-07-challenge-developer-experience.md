Context Enrichment on Messages\n379\t\n380\t### The Problem\n381\t\n382\tWhen a message says \"Submission SUB-001 transitioned to triaged,\" the actor receiving it needs context: What's in the submission? Who's the insured? What carrier? What product? Without context, the actor (human or associate) must make round-trips to fetch data before they can act.\n383\t\n384\t### Three Patterns\n385\t\n386\t**Pattern A: Thin Events (Event Notification)**\n387\t- Message contains only: entity_id, entity_type, operation, timestamp\n388\t- Consumer fetches what it needs via API/CLI\n389\t- Pro: Minimal message size, always fresh data\n390\t- Con: N+1 query problem, requires source to be available, slow for associates\n391\t\n392\t**Pattern B: Fat Events (Event-Carried State Transfer)**\n393\t- Message contains the full entity state + related entities\n394\t- Consumer has everything it needs\n395\t- Pro: Self-contained, no round-trips\n396\t- Con: Large messages, stale data risk, coupling between producer and consumer schema\n397\t\n398\t**Pattern C: Enriched Events (the hybrid)**\n399\t- Message generated as thin event\n400\t- Enrichment step resolves related entities based on routing destination's needs\n401\t- Consumer receives message with relevant context attached\n402\t- Pro: Right-sized context per consumer, decoupled enrichment\n403\t- Con: Enrichment adds latency, must define what to resolve per role\n404\t\n405\t### Recommendation: Pattern C — Enriched Events with Role-Based Context Rules\n406\t\n407\tThe OS already has the concept of \"context scoping\" per role (from the core primitives design). The insurance domain changes slowly (submissions take hours/days, not milliseconds). Emit on business-meaningful changes only.\n483\t\n484\t2. **No distinction between commands and events.** Commands (\"create this policy\") and events (\"policy was created\") are fundamentally different. Commands can fail and be rejected. Events are facts. Mixing them leads to architectures that look event-driven but behave like synchronous RPC.\n485\t\n486\t3. **Debugging distributed message flows.** The number one complaint in production event-driven systems. Old messages in queues may not match new consumers. Data and behavior live together. The field definitions are standard Pydantic, which most Python developers already know. The state_machine dict is readable -- you can glance at it and understand the lifecycle. The @exposed decorator is clean and solves the \"which methods are public\" problem without XML or annotations bloat.\n\nThe auto-generation story (one class definition produces API routes, CLI commands, and skill documentation) is the strongest part of the design. If it works as described, it eliminates the most tedious part of building CRUD applications.\n\n**What would confuse developers:**\n\nThe Beanie specifics are real framework lock-in, regardless of how the documents frame it. Beanie's `Link` type for relationships is not standard Python or Pydantic.
