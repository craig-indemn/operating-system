# Kernel vs. Domain — The OS Primitive Set

## The Key Distinction

When you build a system on the OS (like GIC Email Intelligence), you create domain entities: Email, Submission, Carrier, Draft. Those are specific to the problem being solved.

But Actor and Role aren't domain entities. You don't "create" the concept of an actor when you build a system. Actors and Roles are the OS itself — the foundational fabric that any system built on the OS operates within.

**OS primitives (the kernel):** Exist before any domain system is built. They're the things you use to build everything else.

**Domain entities:** Created per system, using the entity framework, operating within the kernel's actor/role/message infrastructure.

Both have CLI access. Both are queryable. Both are self-describing. But kernel primitives exist at a different level — they ARE the platform.

## Decision: Start Bare, Add Domain Primitives Later

The kernel starts with only the primitives needed to build any system. If insurance-specific domains prove to be universal enough to become primitives, they can be promoted later. The kernel doesn't start with insurance assumptions.

## The Kernel Primitives

| Primitive | What It Is | Why It's Kernel |
|-----------|-----------|-----------------|
| **Organization** | Multi-tenancy scope — the container everything lives in | Without it, nothing is scoped. Universal to any multi-tenant system. |
| **Entity** (framework) | How domain data is defined — structure, lifecycle, CLI, API, skills | The framework itself, not any instance. It's how you build things. |
| **Message** | The nervous system — generated by entity changes, delivered via watches | How changes propagate through the system. Universal to any reactive system. |
| **Actor** | A participant — human or automated — that processes triggers via CLI | Who participates. Universal to any system with users or automation. |
| **Role** | Permissions + watches — defines how actors participate | What actors can do and what comes to them. Universal to any access-controlled system. |
| **Assignment** | Entity ↔ Actor relationship with role context | How specific items are owned by specific actors. Universal to any work management. |

Domain entities (Submission, Email, Carrier, Draft, Policy, etc.) are created USING the entity framework and operate WITHIN the kernel's infrastructure.

## How Things Link Together

### Actor → Role
- An actor holds one or more roles within an organization
- `indemn actor add-role jc@gicunderwriters.com --role underwriter`
- An actor with multiple roles sees the union of all their roles' watches in their queue

### Role → Entity Types (Permissions)
- A role defines which entity types it can read/write and which operations it can perform
- `indemn role set-permissions underwriter --read "submission,quote,email,carrier" --write "submission,quote,draft"`

### Role → Entity Events (Watches)
- A role defines which entity changes generate messages for actors with that role
- `indemn role add-watch underwriter --entity Assessment --on created --when "needs_review=true"`
- The wiring IS the set of watches across all roles

### Entity → Actor (Assignment)
- Any entity can be assigned to one or more actors with a role context
- `indemn assign SUB-001 --to jc@gicunderwriters.com --as underwriter`
- `indemn assign SUB-001 --to compliance@gicunderwriters.com --as compliance_reviewer`
- Assignment is an OS-level relationship, not a field on each entity

### Message → Role + Optional Actor (Targeting)
- Messages target a role by default (any actor with that role can claim)
- If the entity has an assignment for the target role, the message targets that specific actor
- Queue query for an actor: "messages where target_role in my roles AND (target_actor_id is null OR target_actor_id is me)"

## Assignment as a Universal OS Capability

Assignment is NOT a field on domain entities. It's a built-in OS relationship.

**Why not a field on entities:**
- Too narrow (single person per field)
- Too fragmented (every entity defines it differently — `assigned_to`, `underwriter`, `reviewer`)
- Mixes domain data with OS-level concerns

**Assignment is: entity + actor + role context.**

```bash
# Single assignment
indemn assign SUB-001 --to jc@gicunderwriters.com --as underwriter

# Multi-assignment (different roles)
indemn assign SUB-001 --to jc@gicunderwriters.com --as underwriter
indemn assign SUB-001 --to compliance@gicunderwriters.com --as compliance_reviewer

# Multi-assignment (same role, collaborative)
indemn assign DRAFT-001 --to maribel@gicunderwriters.com --as reviewer
indemn assign DRAFT-001 --to jc@gicunderwriters.com --as reviewer

# Query assignments
indemn assignment list --actor jc  # Everything JC owns across all entity types
indemn assignment list --entity SUB-001  # Who owns SUB-001 and in what capacity
```

**Messaging integration:**
When a message is generated about SUB-001 for the "underwriter" role, the OS checks: is there an actor assigned to SUB-001 as underwriter? If yes, target them specifically. If no, target the role generally (any underwriter can claim).

**This gives:**
- Multi-assignment — multiple actors, different or same role contexts
- Universal semantics — works the same for any entity type
- Queryable — see everything an actor owns, or who owns a specific entity
- Visible — actor's queue shows unassigned items for their role + items specifically assigned to them

## Actor Identity

Actors are created at the OS level. Their identity is managed by the OS, not by any domain system.

```bash
# Human actors
indemn actor create --name "JC" --email jc@gicunderwriters.com --type human
indemn actor create --name "Maribel" --email maribel@gicunderwriters.com --type human

# Associate actors
indemn actor create --name "Email Classifier" --type associate --role classifier --skills email-classification --mode hybrid
indemn actor create --name "Submission Linker" --type associate --role linker --skills submission-linking --mode hybrid
```

The actor ID comes from the OS. That identity is referenced by assignments, messages, and audit trails across every system built on the OS.

JC isn't a GIC Email Intelligence concept. JC is an actor in the OS. When the GIC system creates Submissions, those entities generate messages that reach JC because of his role's watches. If a CRM is also built on the OS, JC might have an "account_manager" role too. Same actor, same queue, showing tasks from both systems.

**That's what makes it an operating system — the actors, roles, permissions, and messaging are the kernel. Applications are the domain entities and the associates that operate on them.**

## The Full Picture

```
Organization: GIC Underwriters
  │
  ├── OS Kernel (always present)
  │   ├── Actors: JC, Maribel, Email Classifier, Submission Linker, ...
  │   ├── Roles: underwriter, operations, classifier, linker, assessor, ...
  │   ├── Watches: (on roles — define what each role sees)
  │   ├── Assignments: (entity ↔ actor relationships)
  │   └── Messages: (flowing based on watches, targeting based on assignments)
  │
  └── Domain: GIC Email Intelligence (built on the OS)
      ├── Entities: Email, Submission, Extraction, Assessment, Draft, Carrier, Agent
      ├── Skills: email-classification, submission-linking, situation-assessment, ...
      └── Scheduled tasks: email-sync (cron), stale-check (cron)
```

The domain system creates entities. The kernel handles everything else — who processes them, how changes propagate, who sees what, who owns what.

## How a New System Gets Built on the OS

1. **Define domain entities** — what data does this system manage? (Email, Submission, etc.)
2. **Define roles** — what types of participants does this system have? (classifier, underwriter, operations)
3. **Configure watches** — what entity changes matter to each role?
4. **Create associates** — what automated actors process messages? With what skills?
5. **Assign humans** — who has which roles?
6. **Schedule infrastructure** — what CLI commands run on timers? (email sync, stale checks)
7. **The system runs** — entities change, messages flow, actors process, the system churns

Steps 1 is domain. Steps 2-6 use kernel primitives. Step 7 is the kernel doing its job.
