## Executive Summary

**One new finding, and strong confirmation of Finding 0.**

Pass 2 confirmed Finding 0 at three independent levels — the finding is not an isolated spec bug, it's a consistent architectural-layer mismatch that propagated from spec → code → infrastructure. The design artifacts say agent execution lives in harness images outside the kernel; the spec embedded agent execution inside the kernel's Temporal Worker and the kernel's API server; the code faithfully implements the spec; the infrastructure (Dockerfile + docker-compose.yml) only builds and starts the kernel image.

Beyond Finding 0, Pass 2 surfaced **one additional structural gap** (Finding 0b: the assistant-as-chat-harness gap — a specific arm of Finding 0 that the comprehensive audit called "Finding 2" but which is architecturally the same root cause), plus **one methodological observation** (the deviation entered at the Phase 2 spec and propagated; no verification pass caught it because no pass cross-referenced at this layer).

**Pass 2 also confirmed that eight other subsystems are architecturally clean** — no layer deviations beyond Finding 0 and its direct consequences.

The Phase 2-3 consolidated spec is the singular source of the Finding 0 deviation. The code faithfully implements what the spec says; the spec deviates from the source design.

## 2. Finding 0, Confirmed

### 2.1 What the source design artifacts say (canonical)

The design is unambiguous on the harness pattern across multiple independent artifacts. From `2026-04-10-realtime-architecture-design.md` Part 4:

> "The harness uses the CLI, not a separate Python SDK. **This is the key design decision.** The CLI is already the universal interface. The harness is 'just another client' of the CLI, calling it via subprocess. No dual interface, no second code path, no framework-specific adapters in the kernel."

And the same artifact specifies three deployable images:

> - `indemn/runtime-voice-deepagents:1.2.0` — voice + deepagents + LiveKit Agents
> - `indemn/runtime-chat-deepagents:1.2.0` — chat + deepagents + WebSocket server
> - `indemn/runtime-async-deepagents:1.2.0` — async + deepagents + Temporal worker

The async-deepagents harness is specifically described at line 460-463 of that artifact: "For async runtimes: subscribing to a Temporal task queue named after the Runtime."

The white paper reinforces this in Service Architecture (§ 7):

> "Harnesses. One image per runtime kind and framework combination. For example, a voice harness or a chat harness. Each connects to the API server through internal networking and uses the CLI for all OS operations."

And makes the trust boundary explicit:

> "Inside the trust boundary: the API Server, Queue Processor, and Temporal Worker. [...] Outside the trust boundary: the Base UI, harnesses, the CLI, and Tier 3 applications. All of these access the system through the API."

`2026-04-11-base-ui-operational-surface.md` ties the assistant to the harness pattern (lines 167, 306, 346-347):

> "the conversation panel is a running harness instance — a real-time actor in the session 5 sense"
> "The assistant panel is a running instance of a chat-kind Runtime (or integrated into an existing runtime). Same deployment model as any real-time harness."

`2026-04-11-authentication-design.md` (lines 463-482) confirms the same:

> "The default assistant is not an independent entity — it is a projection of the user into a running actor. **Its harness authenticates using the user's session JWT (injected at session start).**"

`2026-04-13-simplification-pass.md` explicitly lists the harness pattern in Layer 3 of the simplified architecture — it survived the "does this earn its complexity?" review intact.

### 2.2 What the consolidated specs say (the deviation source)

**The deviation enters the pipeline in the Phase 2-3 consolidated spec.**

Phase 0-1 spec (lines 145-148) is consistent with the design — `kernel/temporal/` is a stub (`client.py` only, "Phase 2 activation"). No agent execution in Phase 0-1. Harnesses "Phase 5" per the deferral list.

Phase 2-3 spec §2.1-§2.4 (lines 58-759) places agent execution inside the kernel:
- `kernel/temporal/worker.py` registers `process_with_associate` as an activity on `task_queue="indemn-kernel"` (the kernel's queue).
- `kernel/temporal/activities.py::process_with_associate` contains the full agent execution loop (`_execute_deterministic`, `_execute_reasoning`, `_execute_hybrid`), with `_execute_reasoning` importing `anthropic` directly and iterating up to 20 times with the `execute_command` tool-use pattern.
- The activity runs inside the kernel Temporal Worker process — inside the trust boundary, with direct MongoDB access.

**The Phase 2-3 spec does not explain this deviation from the harness-pattern design and does not flag it.** No "this is temporary, will move to harness in Phase 5" note exists.

Phase 4-5 spec has two layers of deviation:
- §4.7 (lines 836-876) places the assistant as a kernel API endpoint at `kernel/api/assistant.py`, streaming text via `anthropic.AsyncAnthropic()` with **no tools** (no `tools=[]` parameter). This is Finding 2 from the comprehensive audit, which Pass 2 recognizes as a direct architectural-layer consequence of Finding 0.
- §5.3 (lines 1587-1734) does show a complete voice harness example at `harnesses/voice-deepagents/main.py`. Per Phase 5 acceptance criteria (line 2051-2053): "HARNESS USES CLI ONLY — All harness-to-kernel communication via CLI subprocess → no direct MongoDB → no Python SDK." But this is ONLY the voice harness, and the comprehensive audit confirmed it was "spec as example code" — not implemented as a deployable artifact.

**No `harnesses/chat-deepagents/` example exists in the Phase 5 spec.** The chat harness that would house the assistant is missing from the spec entirely. `harnesses/async-deepagents/` — the harness that would house `process_with_associate` — is also missing from the Phase 5 spec.

Phase 6-7 spec is operational (entity/skill/rule configuration) and adds no architectural changes. It inherits Finding 0.

### 2.3 What the code does

The Phase 2-3 spec was implemented faithfully:

- **`kernel/temporal/activities.py`** (633 lines) — the agent execution loop runs here as a Temporal activity. Line 376 imports `anthropic`. Lines 370-487 contain `_execute_reasoning` with the LLM tool-use loop. Lines 328-367 contain `_execute_deterministic`. Lines 490-500 contain `_execute_hybrid`.

- **`kernel/api/assistant.py`** (107 lines) — the assistant endpoint. Line 37 imports `anthropic`. Lines 50-57 call `client.messages.stream(...)` with NO `tools=[]` parameter. The system prompt says "You can execute any CLI command the user has permission for" but the LLM has no mechanism to do so.

- **`kernel/temporal/workflows.py`** (255 lines) — `ProcessMessageWorkflow` orchestrates `process_with_associate` at line 68. `HumanReviewWorkflow` and `BulkExecuteWorkflow` are also here (these two are legitimately kernel-side).

- **`kernel/temporal/worker.py`** (82 lines) — registers all three workflows and all 8 activities on `task_queue="indemn-kernel"`. The worker process runs as `indemn-temporal-worker` Railway service per Phase 0-1 spec.

- **No `harnesses/` directory exists in the repo.** `find /Users/home/Repositories/indemn-os -type d -name "harness*"` returns empty. The Phase 5 spec's voice harness example was not implemented.

- **`Dockerfile`** (18 lines) — builds only the kernel image. Copies `kernel/`, `kernel_entities/`, `seed/`. No harness image definitions.

- **`docker-compose.yml`** (36 lines) — starts `indemn-api` and `indemn-queue-processor` only. No harness services, no Temporal Worker service for local dev.

### 2.4 Severity, impact, cascade

Finding 0 is **STRUCTURAL** and cascades across layers:

| Layer | State |
|---|---|
| Source design (12 artifacts) | Consistent: harness pattern with 3 deployable images, LLM-agnostic kernel, CLI as universal interface |
| Spec (Phase 2-3) | **Deviation**: `process_with_associate` as kernel activity with direct `anthropic` import |
| Spec (Phase 4-5) | **Deviation**: assistant as kernel API endpoint; voice harness as example code (only); no chat-harness, no async-harness |
| Code | Faithfully implements the deviated spec |
| Infrastructure | Only kernel image built; no harness services configured |

**Direct consequences (all traced to the same root):**

1. **Assistant has no tools** (comprehensive audit Finding 2). The kernel endpoint was spec'd without tools; the harness-pattern-based chat assistant would naturally have them.

2. **Kernel knows about Anthropic.** Two places (`kernel/temporal/activities.py`, `kernel/api/assistant.py`) import `anthropic` directly. Design principle is LLM-agnostic kernel. Swapping LLM providers requires kernel changes.

3. **No real-time channels work.** Chat and voice harnesses don't exist; the kernel's infrastructure (events stream, attention heartbeat, interaction transfer — all correctly specified and implemented) has no consumers.

4. **Unified actor model partially broken.** Async associate execution runs in-kernel (Temporal activity); real-time doesn't run at all. They were supposed to share code via the harness pattern.

5. **Sandbox execution missing.** Per Layer 3 associate system design, the agent runs in Daytona (or equivalent) for isolation. With agent execution in the kernel Temporal worker, there's no sandbox — the agent runs with full DB credentials.

6. **Temporal task-queue model flattened.** Per design, each Runtime has its own task queue; harnesses subscribe by Runtime. Current implementation has one task queue (`"indemn-kernel"`) for everything.

### 2.5 How this finding wasn't caught

Per the 2026-04-15 comprehensive audit's §3a (Audit Methodology Gap), no prior pass cross-referenced consolidated specs back to source design artifacts at the architectural-layer level. The spec consolidation, the 90-gap review, the 4-pass verification, the shakeout, and the Pass 1 audit all took prior layers as authoritative. Pass 2 closed that gap — and confirmed the exact finding the earlier methodology would have missed.

## 3. Pass 2's Additional Finding: Finding 0b (Scope Clarification)

### 3.1 The assistant-as-chat-harness gap, formalized

The comprehensive audit called this "Finding 2: Assistant has no tools." Pass 2 clarifies that this is not a separate finding but **a specific arm of Finding 0** — the chat-harness gap.

The root cause: Phase 4 needed the assistant to work, but the harness pattern wasn't in Phase 4's scope (it's Phase 5). Phase 4 improvised with a kernel API endpoint. Then Phase 5 specified the harness pattern — but only demonstrated voice. The chat harness that would have closed the loop was missing from the spec.

**Three things are true simultaneously and together make the finding:**

1. `kernel/api/assistant.py` exists as a kernel API endpoint that streams text from Anthropic.
2. That endpoint has no tools — the LLM cannot execute anything, despite the system prompt claiming it can.
3. The design says the assistant should be a running instance of a chat-kind Runtime (a harness) that uses the CLI for operations — confirmed by THREE independent design artifacts (base-ui-operational-surface, authentication-design, realtime-architecture-design).

The fix for Finding 2 (add tools to the endpoint) would address the symptom but not the root cause. The proper fix is to build `harnesses/chat-deepagents/` and delete `kernel/api/assistant.py`.

### 3.2 The async-harness gap

Parallel to the chat gap: the `indemn/runtime-async-deepagents` image doesn't exist and wasn't specified in Phase 5. The async agent execution path (`kernel/temporal/activities.py::process_with_associate`) has no migration target in the spec.

Pass 2 confirms the missing harness images:

| Harness | In spec? | In code? |
|---|---|---|
| `indemn/runtime-voice-deepagents` | Example only (Phase 5 §5.3) | Not built |
| `indemn/runtime-chat-deepagents` | **Not specified** | Not built |
| `indemn/runtime-async-deepagents` | **Not specified** | Not built |

All three are required by the realtime-architecture-design artifact. Only voice exists as example code; chat and async are missing from the spec entirely.

### 4.4 Authentication — CLEAN

**Source**: `2026-04-11-authentication-design.md` says "Kernel handles mechanics (validation, session lifecycle, token issuance, MFA challenge flow)."

**Code**: `kernel/auth/session_manager.py`, `kernel/auth/jwt.py`, `kernel/auth/middleware.py`, `kernel/auth/password.py`, `kernel/auth/rate_limit.py`, `kernel/auth/audit.py`, `kernel/auth/token.py`, `kernel/api/auth_routes.py`, `kernel/api/registration.py` — all correctly kernel-side.

### 4.5 Bulk operations — CLEAN

**Source**: `2026-04-13-simplification-pass.md` explicitly moved watch coalescing OUT of the kernel. No `coalesce` field on watches, no `batch_id` on messages, no grouping logic in emission path. UI groups by correlation_id.

## 5. Methodological Observations

### 5.1 Why the deviation entered the Phase 2-3 spec

Based on phasing analysis:

- Phase 2 (Associate Execution) comes before Phase 5 (Real-Time, harness pattern).
- Phase 2 needed agent execution to work for the phase to be buildable.
- The harness pattern wasn't formally specified until Phase 5.
- The Phase 2 spec writer placed agent execution inside the existing kernel Temporal Worker — the nearest available execution environment.
- No migration note, no "to be moved to harness in Phase 5," no TODO.

**The spec should have either:**
- Reordered phases (Phase 5 harness pattern before Phase 2 associate execution), OR
- Built the async-deepagents harness image in Phase 2 as a deployable (establishing the harness pattern early), OR
- Explicitly flagged the placement as temporary with a migration plan.

None of these happened.

### 5.2 Why no prior audit caught it

Per the 2026-04-15 comprehensive audit's §3a:

| Audit | What it checked | What it missed |
|---|---|---|
| Spec consolidation (90 gaps) | Are all mechanisms specified? | Architectural-layer placement |
| Verification report (4 passes, 29 findings) | Do specs match white paper? | Whether specs drifted from source design |
| Shakeout (19 findings) | Does the system boot? | Whether code is in the right place |
| Pass 1 audit (initial) | Does code match spec? | Whether spec matches source design at layer level |

Every pass took the prior layer as authoritative. **No pass cross-referenced consolidated specs back to source design artifacts at the architectural-layer level.**

Pass 2 was explicitly designed to close this gap. It did. It found Finding 0 exactly where the methodology predicted — at the consolidated-spec-to-source-design interface, specifically in Phase 2-3 §2.4 Activities and Phase 4-5 §4.7 The Assistant.

### 5.3 Why the systematic-audit skill matters

The 2026-04-15 audit skipped or partially-read multiple files despite explicit instructions to read everything. Stronger instructions did not fix it. Pass 2 used a script-enforced manifest to remove reading-decision agency; the script picked the next file, the validator refused to mark files checked without structured notes, the synthesis-gate hook blocked report writing until the manifest was complete.

All 41 files in the Pass 2 manifest were read in full. Notes for each file were validated. The matrix was built from the notes. The report was written after the gate allowed it. The audit trail is verifiable at `.audit/manifest.yml` + `.audit/notes/` + `.audit/cross-reference-matrix.md`.
