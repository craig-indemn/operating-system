# Core Primitives Architecture

## Status: ACTIVE DESIGN — Refining toward first principles

This artifact captures the emerging architecture for the OS's core primitives. The goal: the simplest possible foundation that composes into everything the OS needs to do.

## The Four Primitives

Everything in the OS is built from four concepts:

### 1. Entity
Data with structure, relationships, and optionally a lifecycle. **One uniform class — no mixins, no type hierarchy.**

- Has typed fields (Pydantic validation)
- Has relationships to other entities (Beanie Link/BackLink)
- Has a lifecycle (state machine activates IF `state_machine` is defined on the class)
- Has permissions (role-based)
- Has a flexible data section (activates IF field references a form schema)
- Persisted in MongoDB, scoped by org_id
- Exposed via API, CLI, auto-generated skill
- Has optimistic concurrency (version field on every entity)
- **Message generation activates IF routing rules exist for this entity type in this org**

All capabilities are built into one Entity class. What activates them is configuration — both on the class (state machine, fields, permissions) and in the system (routing rules, form schemas). Inactive capabilities have zero overhead (in-memory cache lookups, nanosecond cost).

A reference table (LineOfBusiness) and a complex entity (Submission) are the same class. Submission has a state machine defined and routing rules configured. LineOfBusiness has neither. But if someone later adds a routing rule for LineOfBusiness changes, it starts generating messages. No code change.

**This is the key to scaling to 1000 companies:** entities are uniform infrastructure. Associates (with per-org skills) are the customization layer. Configuration (routing rules, product schemas, associate skills) determines per-org behavior. Code changes are for new CAPABILITIES, not new CUSTOMERS.

**NOTE:** This supersedes the mixin-based approach in the Layer 1 design doc (2026-03-30). No StateMachineMixin, EventMixin, MessageMixin as separate mixins. One Entity class with uniform capabilities, configuration-driven activation.

### 2. Message
The nervous system. Generated by entity changes, routed to actors.

- Triggered by: entity creation, update, deletion, state transition, field change, relationship change
- Contains: what changed, which entity, the relevant context (attached entities)
- Routed to: one or more actors based on routing rules defined by their role
- Has priority, ordering, aging
- Escalates if unprocessed (configurable per message type)

Messages unify everything that currently requires separate systems:
- **Notifications** = messages with a notification type
- **Tasks** = messages routed to a human or associate queue
- **Triggers** = messages that activate an associate
- **HITL** = messages routed to a human queue instead of an associate
- **Workflow steps** = messages generated by the previous step's entity change
- **Channel input** = messages from email/voice/chat entering the system
- **Events** = messages between sub-domains (what we previously called RabbitMQ events)

A message is NOT an entity (it doesn't have a lifecycle and CRUD). It's a lower-level primitive — a signal that flows through the system.

### 3. Actor
Any participant in the system — human or associate. Has a role. Has a queue.

- An actor is either a human user or an associate (AI agent)
- Identified by their role within an organization
- Has a **queue** — their work inbox of messages needing their attention
- Processes messages: reads the message, acts on it (changes entity state), which generates new messages
- The queue IS the actor's view of the system — what needs their input right now

For humans: the queue is their home screen / dashboard. What's waiting on me?
For associates: the queue is the trigger mechanism. When a message arrives, the associate is invoked.

The same assignment model works for both: "assign this to an underwriter" = route the message to actors with the underwriter role.

### 4. Role
Defines what an actor can do AND what comes to them. One concept, two functions.

- **Permissions**: which entities the role can read/write, which operations it can perform
- **Routing**: which messages get delivered to actors with this role
- **UI generation**: the interface auto-generates based on what the role can access
- **CLI scoping**: the CLI shows commands the role has permission for
- **Skill scoping**: associates with this role get skills for entities they can access
- **Context scoping**: what entities are pre-loaded as context for associates in this role

A role is not just access control — it's the complete definition of how an actor participates in the system.

## How It All Works Together

```
Entity changes (any change — create, update, state transition)
    → Messages generated (what changed, context attached)
    → Messages routed to actors based on role routing rules
    → Actor's queue receives the message
    
    If actor is human:
        → Message appears in their UI queue / dashboard
        → Human acts on it (via UI, CLI, or AI assistant)
        → Action changes entity state
        → New messages generated → cycle continues
    
    If actor is associate:
        → Message triggers the associate
        → Associate receives message + attached entity context
        → Associate reasons + executes CLI commands
        → Commands change entity state
        → New messages generated → cycle continues
```

The system is always churning. Every entity change propagates through the system as messages, reaching the right actors, who process them and generate more changes. Nothing falls through — everything is in someone's (or something's) queue.

## Context Management (Entity Attachment)

When a message is generated, the system automatically resolves and attaches related entities as context:

```
Submission SUB-001 transitions to "triaged"
    → Message generated
    → System resolves context:
        - The submission itself (full data)
        - The insured (via submission.insured relationship)
        - The carrier (via submission.carrier)
        - The product (via submission.product, including form_schema)
        - Recent interactions on this submission
        - Related documents/attachments
    → Context attached to the message
    → Message routed to the intake associate
    → Associate receives rich context — can reason immediately without CLI round-trips
```

This is configurable per role/message type — how deep to resolve, which relationships to follow, how much history to include. Balances richness with context window limits.

For associates: reduces CLI round-trips, improves reasoning quality, speeds up processing.
For humans: their UI view shows the full context of each queue item — click on a submission and see everything related.

This is analogous to how Claude Code pre-loads CLAUDE.md and skills — the system curates context before the actor starts working.

## What This Unifies (Previously Separate Concepts)

| Previous Concept | Now | How |
|-----------------|-----|-----|
| Notifications | Messages | Route to human queues with notification type |
| Tasks | Messages | Route to actor queues with task type and priority |
| HITL / Drafts | Messages | Route review requests to human queues |
| Workflow steps | Messages | State change → message → next actor → state change → ... |
| Event handlers | Messages | Cross-domain routing replaces explicit event subscriptions |
| Associate triggers | Messages | Associate queue receives messages matching its role |
| Channel input | Messages | Email/voice/chat arrive as messages in the system |
| Work assignment | Messages | "Assign to X" = route message to X's queue |
| Escalation | Messages | Unprocessed message re-routes after timeout |

## Workflows Emerge — Not Orchestrated

With the messaging model, explicit workflow orchestration (workflow entity, workflow runner) may be unnecessary. Workflows emerge from connected entity state machines:

**Binding "workflow":**
1. Quote transitions to "accepted" → message routes to binding actor (service or associate)
2. Binding actor creates Policy → Policy created message routes to payment actor
3. Payment actor initiates payment → Payment completed message routes to certificate actor
4. Certificate actor generates certificate → Certificate created message routes to notification actor
5. Notification actor sends confirmation to retail agent

No workflow runner. No workflow definition. Just entities with state machines, messages, and routing. Each step is an actor processing a message, changing state, and generating the next message.

**But** — some workflows may still benefit from explicit definition for:
- Complex conditional logic (if X then route to A, else route to B)
- Parallel fan-out (route to 5 carriers simultaneously)
- Timeout/escalation rules (if no response in 3 days, escalate)
- Audit/compliance (need to prove the workflow followed a defined process)

The Workflow entity might evolve from "orchestration engine" to "routing configuration" — defining the rules for how messages flow between actors for a specific process, rather than executing steps.

## The Actor's Personal AI Assistant

Every human actor can have a personal associate — an AI assistant that has the same role permissions and sees the same queue. The human can:
- Ask the assistant to handle queue items: "Process these routine certificate requests"
- Ask for analysis: "Which submissions in my queue are highest priority?"
- Delegate: "Draft responses for all the info requests, I'll review them"
- Get briefings: "What happened overnight?"

The assistant IS an associate configured with the same role. It reads the same entity skills. It operates through the same CLI. It's just another actor in the system — one that works alongside the human.

## Real-Time Visibility

Because everything flows through messages:
- **System health** = message flow rate + queue depths + processing times
- **Bottlenecks** = queues that are growing (someone is behind)
- **Progress** = entity state distribution (how many submissions at each stage)
- **History** = complete message trail for any entity (audit trail)
- **Activity** = who's processing what right now

The monitoring dashboard IS the message flow visualization. No separate monitoring system.

## UI Auto-Generation from Roles

A role defines what entities you can access → the UI generates from that:

- **Queue view** (home screen): your pending messages, prioritized
- **Entity views**: tables and forms for entities your role can access
- **Actions**: buttons for operations your role can perform (@exposed methods)
- **Context panels**: related entities shown alongside the primary entity

Different roles see different views — not because we built separate UIs, but because the role determines which entities and messages are visible.

An underwriter's UI shows: their submission queue, submission detail forms, quote comparison views, approval/decline actions.
A CSR's UI shows: their customer inquiry queue, policy lookup, certificate generation, communication drafts.
Same UI framework, different role configuration.

## Foundation Hardening (from adversarial review, 2026-04-07)

Five independent reviewers (insurance practitioner, distributed systems engineer, developer experience, real-time systems, startup CTO) challenged this architecture. The primitives held up. The implementation details needed hardening. Below are the non-negotiable additions to the foundation.

### 1. Optimistic Concurrency — Version Field on Every Entity

Without this, concurrent writes silently corrupt data. Two actors processing messages about the same entity overwrite each other's changes with no error.

```python
class Entity(Document, ...):
    version: int = 1
    
    async def save(self, **kwargs):
        # Conditional write — fails if version changed since read
        result = await self.get_motor_collection().update_one(
            {"_id": self.id, "version": self.version},
            {"$set": self.dict(), "$inc": {"version": 1}}
        )
        if result.modified_count == 0:
            raise ConcurrencyConflict(f"{self.__class__.__name__} {self.id} was modified by another actor")
```

Every actor reads the version, does its work, and writes with a version guard. If the version changed, the write fails and the actor re-reads and re-evaluates. This is table stakes for a multi-actor system.

### 2. Transactional Entity+Message Atomicity

Entity save and message creation MUST be in the same MongoDB transaction. Otherwise, if the process crashes between entity save and message creation, the entity changes but nobody gets notified.

```python
async def save_with_messages(entity):
    async with await client.start_session() as session:
        async with session.start_transaction():
            await entity.save(session=session)
            messages = await evaluate_routing_rules(entity)
            for msg in messages:
                await msg.insert(session=session)
            # Both succeed or both fail — no orphaned state changes
```

Cost: 5-15% write latency increase. Benefit: eliminates the most dangerous failure mode (silent message loss).

### 3. Message Infrastructure — Decided

**MVP: MongoDB only.** Messages stored in a `messages` collection. `findOneAndUpdate` for atomic claiming. Polling for consumption (5-second intervals, adaptive). No separate broker.

**Why:** One database, one transaction boundary. Entity+message atomicity is trivial with MongoDB transactions. No dual-write problem. Messages are queryable (queue = filtered find). Audit trail built in.

**Scale path (additive, nothing ripped out):**
- Add Change Streams as wake-up signal when polling latency matters
- Add Redis Pub/Sub for real-time UI push
- Add RabbitMQ as notification relay when volume exceeds polling capacity
- MongoDB stays as source of truth for messages

### 4. Message Schema — Decided

```python
class Message(Document):
    # Identity
    org_id: Indexed(ObjectId)
    entity_type: str
    entity_id: ObjectId
    event_type: str              # "status_changed", "created", "field_changed"
    
    # Routing
    target_role: Indexed(str)
    target_actor_id: Optional[ObjectId]  # For re-routes, escalations
    
    # Traceability (from day one — impossible to retrofit)
    correlation_id: str          # Traces the entire cascade back to originating change
    causation_id: Optional[str]  # The message that directly caused this one
    depth: int = 0               # Cascade depth — circuit breaker at max depth
    
    # State
    status: Literal["pending", "processing", "completed", "failed", "escalated", "dead_letter"]
    claimed_by: Optional[ObjectId]
    claimed_at: Optional[datetime]
    visibility_timeout: Optional[datetime]  # Stuck message recovery
    attempt_count: int = 0
    max_attempts: int = 3
    
    # Priority and aging
    priority: Literal["critical", "high", "normal", "low"] = "normal"
    created_at: datetime
    due_by: Optional[datetime]   # Escalation deadline
    
    # Context (inline for associates, shallow for humans)
    context: dict = {}           # Resolved entity data
    changes: dict = {}           # What changed
    
    # Completion
    completed_at: Optional[datetime]
    result: Optional[dict]

    class Settings:
        name = "messages"
        indexes = [
            [("org_id", 1), ("target_role", 1), ("status", 1), ("priority", -1), ("created_at", 1)],
            [("entity_type", 1), ("entity_id", 1), ("created_at", -1)],  # Audit trail
            [("status", 1), ("visibility_timeout", 1)],  # Stuck message recovery
            [("status", 1), ("due_by", 1)],  # Escalation sweep
            [("correlation_id", 1), ("created_at", 1)],  # Cascade debugging
        ]
```

### 5. Visibility Timeout — Stuck Message Recovery

When an actor claims a message, it gets a visibility timeout (e.g., 5 minutes for associates, 24 hours for humans). If the actor crashes or fails to complete, the message becomes claimable again after the timeout.

```python
# Claim query includes timed-out messages
message = await Message.get_motor_collection().find_one_and_update(
    {
        "target_role": actor.role,
        "org_id": actor.org_id,
        "$or": [
            {"status": "pending"},
            {"status": "processing", "visibility_timeout": {"$lt": utcnow()}}
        ]
    },
    {
        "$set": {
            "status": "processing",
            "claimed_by": actor.id,
            "claimed_at": utcnow(),
            "visibility_timeout": utcnow() + timedelta(minutes=5)
        },
        "$inc": {"attempt_count": 1}
    },
    sort=[("priority", -1), ("created_at", 1)],
    return_document=True,
)
```

After `max_attempts` exceeded → status moves to `dead_letter` for investigation.

### 6. Cascade Depth Tracking + Circuit Breaker

Every message inherits depth from its parent. Hard stop at configurable max depth (default 10). Prevents infinite loops from misconfigured routing.

```python
async def _generate_messages(self, event_type, changes, parent_message=None):
    depth = (parent_message.depth + 1) if parent_message else 0
    correlation_id = parent_message.correlation_id if parent_message else str(uuid4())
    
    if depth > MAX_CASCADE_DEPTH:
        log.warning(f"Cascade depth exceeded: correlation={correlation_id}")
        # Create circuit-broken message for investigation
        await Message(status="circuit_broken", depth=depth, ...).insert()
        return
    
    # Normal message creation with inherited traceability
    for rule in matching_rules:
        await Message(
            depth=depth,
            correlation_id=correlation_id,
            causation_id=parent_message.id if parent_message else None,
            ...
        ).insert()
```

### 7. Idempotent Message Processing

Because visibility timeouts allow re-delivery, all message handlers MUST be idempotent. The framework provides utilities:

```python
async def handle_message(msg: Message):
    # Check if already processed
    if await ProcessedLog.find_one({"message_id": msg.id}):
        return  # Already handled
    
    # Check if entity moved past expected state
    entity = await Entity.get(msg.entity_id)
    if entity.version > msg.context.get("entity_version"):
        return  # Entity already advanced
    
    # Process...
    
    # Record completion
    await ProcessedLog(message_id=msg.id).insert()
```

### 8. Routing Rules — Decided

RoutingRule as a separate entity, configurable per organization via CLI. NOT hardcoded on entity classes.

```python
class RoutingRule(Entity):
    sub_domain = "platform"
    
    entity_type: str           # "Submission", "Policy"
    event: str                 # "status_changed", "created", "@submit_to_carrier"
    conditions: dict           # {"to_status": "triaged", "lob": "GL"}
    target_role: str           # "intake_associate", "underwriter"
    context_depth: str         # "none", "shallow", "deep"
    priority: str = "normal"
    sla_hours: Optional[int]   # Escalation deadline
```

Rules cached in memory per org (60-second TTL). For complex routing logic beyond field matching, a code extension point allows Python functions registered as custom matchers.

### 9. Selective Message Emission

NOT every field change generates a message. Only:
- State machine transitions (status changes)
- @exposed method invocations (business operations)
- Entity creation and deletion
- Explicitly marked fields (opt-in, not opt-out)

This prevents drowning in noise from routine field updates.

### 10. MessageBus Abstraction

The messaging infrastructure is behind an interface. MongoDB implements it now. Swappable later.

```python
class MessageBus(Protocol):
    async def publish(self, message: Message) -> None: ...
    async def consume(self, role: str, org_id: ObjectId) -> AsyncIterator[Message]: ...
    async def claim(self, message_id: ObjectId, actor_id: ObjectId) -> Optional[Message]: ...
    async def complete(self, message_id: ObjectId, result: dict) -> None: ...
```

Cost: one hour of design. Prevents painful migration later.

## Real-Time Channels — NOT a Separate Architecture

The adversarial review initially proposed a separate "Interaction Host" pattern with its own state management, Redis Streams, and Entity Sync Worker. After further analysis: **this was over-engineering. Real-time and async use the same associate pattern.**

An associate is a deep agent that reads skills and executes CLI commands. This is true whether it's processing an email (async, one-shot) or handling a phone call (real-time, multi-turn conversation). The mechanism is identical. The difference is the trigger and the I/O duration.

| | Async | Real-Time |
|---|---|---|
| **Trigger** | Message in queue | Channel opens (call/chat) |
| **Context** | Attached to message | Pre-loaded at connection start |
| **Processing** | Deep agent reads skills, executes CLI | Same |
| **Entity operations** | CLI → entity saves → messages | Same |
| **I/O** | One-shot | Streaming multi-turn |
| **Completion** | Message marked complete | Channel closes, Interaction resolved |

**One implementation detail for voice latency:** entity saves during voice calls should be non-blocking (`asyncio.create_task(entity.save())`). The associate continues the conversation. The save + message generation happens in the background. If a save fails (version conflict), the associate discovers it next turn. This is a one-line call-site decision, not a separate architecture.

**What exists:**
- **Interaction entity** — tracks the conversation (started → active → resolved → closed). Normal entity.
- **Pre-loaded context** — entity graph resolved before the agent starts. Useful for BOTH async and real-time.
- **Channel adapters** — voice (LiveKit/Twilio), chat (WebSocket), SMS (Twilio) provide I/O. The associate doesn't know which channel it's on.

**What was proposed and dropped:**
- Separate Interaction Host class → unnecessary, deep agent framework handles conversation state
- Message suppression during interactions → wrong, entity changes should route normally
- Entity Sync Worker → unnecessary, entity saves are fast enough (3-10ms reads, non-blocking writes)
- Redis Streams for coordination → unnecessary, no separate system to coordinate with
- "System of engagement" vs "system of record" → one system, channels are just I/O

**Real-time UI updates** (WebSocket push when entities change) is a separate UI-layer concern handled by Redis Pub/Sub or MongoDB Change Streams. Not related to how associates work.

## Build Order — Revised

The adversarial review consensus: **build entities by hand first, then extract the framework.**

1. Build 3-5 representative entities manually (Submission, Policy, Contact, Associate, Organization) — proves the patterns work
2. Extract the base Entity class and mixins from the working entities
3. Parallelize remaining entities using the proven framework
4. Add the messaging system (can be developed alongside the first entities)
5. Add the real-time layer when voice/chat integration is needed

This inverts the original plan (framework first → entities in parallel). The framework emerges from real usage, not from theory.

## Relationship to Previous Design

This primitives model refines but doesn't contradict layers 1-5:
- **Layer 1 (Entity Framework)**: Stays — Beanie + Pydantic + MongoDB. HARDENED with version field, transactions, selective emission.
- **Layer 2 (API/CLI/Skills)**: Stays — auto-generation from entities. PLUS: observability CLI commands for messaging infrastructure.
- **Layer 3 (Associates)**: Stays — deep agents with CLI sandbox. Now: triggered by messages with pre-loaded context. Handlers must be idempotent.
- **Layer 4 (Integrations)**: Stays — adapter pattern on entity operations. External input arrives as messages.
- **Layer 5 (Experience)**: Evolves — UI driven by role + queue. Real-time via Interaction Host + Redis Pub/Sub + WebSocket. NOT Change Streams for fan-out.

What changes is the CONNECTIVE TISSUE between the layers. Previously: events as a side mechanism. Now: messages as THE mechanism, hardened with transactions, traceability, stuck recovery, cascade protection, and idempotency.

Plus: the real-time layer is a first-class architectural component, not a bypass mode.
