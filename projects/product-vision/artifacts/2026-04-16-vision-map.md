## 1. The Thesis (foundational — craigs-vision.md + vision v1/v2 + white-paper)

- **"We are modeling the businesses in our space as entities and models and integrating AI into the entire system."**
- **Object-oriented insurance system**: every insurance concept has schema + state machine + API + CLI.
- **AI-native from the ground up.** AI agents are a CHANNEL into the platform, not a separate system.
- **The factory thesis** (vision v2): EventGuard proved AI can run an insurance business (2,934 policies, $766K premium, $2.78M acquisition). The factory makes it repeatable in weeks, not months.
- **Moat = insurance-native domain model + agent-accessible platform + compounding intelligence across every deployment.**
- **CLI-first, agent-first**: if it's in the CLI, an agent can do it. If an agent can do it, it scales infinitely.

## 5. Kernel Mechanisms (all in `kernel/`)

### 5.1 save_tracked() — #1 architectural invariant
Entity write + changes record + watch evaluation + message creation — **ONE MongoDB transaction**. (Regressed in shakeout 2026-04-15; fixed 2026-04-16 per comprehensive audit. Verified by Pass 2.)

### 5.2 Selective emission (one-save-one-event discipline)
Only state transitions + @exposed methods + create/delete emit messages. NOT every field change. @exposed method completion is the emission boundary. (Ironing round 3.)

### 5.3 Watch evaluation
- Cached (60s TTL), shared evaluator with rules (JSON conditions).
- **Entity-local only** — no cross-entity reads during evaluation (pressure-test finding #3).
- Scope qualifier: `field_path` (ownership scoping) or `active_context` (real-time routing via Attention lookup). Resolved at emit time → `target_actor_id` on message.
- Cache invalidation on Role save (shakeout Finding 10 fix).
- **NO coalescing in kernel** (simplification-pass removed it). Coalescing is UI-only, grouping by correlation_id.

### 5.4 Rule engine + Lookups
- Rules: condition → `set_fields` OR `force_reasoning` action.
- Lookups: shared reference data, bulk-importable CSV, separate from rules (capabilities-model-review finding).
- Veto rules: force_reasoning override of positive matches.
- Rule groups with lifecycle (Draft → Active → Archived); evaluator checks group status.

### 5.5 Capability library + `--auto` pattern
- Kernel capabilities ARE entity methods (ironing round 2 unification). No separate concept.
- `--auto` pattern: try configured rules deterministically; if `needs_reasoning: true`, LLM fallback via associate skill.
- Auto-classify, auto-link, completeness-check, fuzzy-search, stale-check, pattern-extract, etc.

### 5.6 Correlation + causation + depth
- correlation_id = OTEL trace_id.
- causation_id = parent message.
- cascade depth with max 10; circuit breaker.
- Bootstrap entity cascade guard at depth 1 (ironing round 3).

### 5.7 Visibility timeout + idempotent processing + max_attempts
- Message claiming: `findOneAndUpdate` with `$or: [pending, processing+timed_out]`.
- Max attempts → dead_letter status.
- Handlers must be idempotent (pressure test hardening #4).

### 5.8 Optimistic dispatch + sweep backstop
- After `save_tracked` commits, API fires-and-forgets Temporal workflow start.
- Queue Processor sweeps every few seconds for un-dispatched messages.
- Replaces outbox-poller proposal.

### 5.9 Organizations as environments
- Clone/diff/deploy copies CONFIG only (no instances, no messages, no secrets).
- YAML export + import (org_lifecycle).
- Per-org EntityDefinitions (verification report finding F-1).

### 5.10 Context propagation
- `X-Correlation-ID` + `X-Cascade-Depth` headers.
- contextvars for org_id + actor_id (gap resolution G-71).

## 6. Entity Framework (everything-is-data)

- **Entity definitions as DATA in MongoDB** (`entity_definitions` collection), not Python class files on disk. Exception: 7 kernel entities are hand-written Python classes.
- Dynamic class creation via `create_model` + Beanie Document inheritance at startup (`kernel/entity/factory.py`).
- Auto-generation of CLI + API + skills per entity (entity registration).
- Flexible data validation (`flexible_data_schema` field).
- Schema migration first-class: rename / add / remove / convert-field with batching + dry-run + aliases + rollback.
- **Dual base class post-shakeout**: `KernelBaseEntity` (Beanie) for 7 kernel entities + `DomainBaseEntity` (Pydantic + raw Motor + `_DomainQuery`) for dynamic domain entities.

- Voice clients for humans = Integration with `system_type=voice_client`, owner=actor.
- Identity providers = Integrations (SSO).

## 8. Authentication (authentication-design)

- **Session as 7th kernel entity.** Types: `user_session`, `associate_service` (for owner-bound scheduled associates).
- **5 auth method types**: password (Argon2id in MongoDB), totp (seed in Secrets Manager), sso (via Integration), token (opaque long-lived, hashed), magic_link (one-time).
- **Hybrid JWT + opaque refresh**: access token 15 min (HS256 with jti); refresh token opaque random, hashed, stored in Secrets Manager. 30s rotation overlap.
- **JWT signing = platform-wide key** (not per-org). Security via org_id in claims.
- **MFA policy**: role-level `mfa_required` + actor `mfa_exempt` override + org `default_mfa_required`. WebAuthn + per-operation MFA deferred.
- **Platform admin cross-org**: `_platform` system org + `PlatformCollection` + work-type tagging + time-limited (4h default, 24h max) + customer-configurable notification.
- **Claims refresh on role change** (not revocation): `claims_stale=true`, next request auto-refreshes.
- **Pre-auth rate limiting**: 5 failures / 10 min → 30 min lockout.
- **Revocation cache**: in-memory per API instance, Change Stream invalidation + bootstrap-from-DB on startup.
- **Default assistant inherits user's Session JWT** (injected into harness at session start) — **per design, this is a chat-harness instance**. Current code has it as kernel endpoint → **Finding 0b**.
- **Owner-bound scheduled associates use own service token** (Session of type `associate_service`).
- **First-org bootstrap**: magic_link token via stdout.
- **Tier 3 self-service signup**: MVP scope is org + first admin + password + email verify + first API key. NOT in MVP: billing, plans, email domain verification, sub-user invitation, OAuth app management.

## 9. Base UI (base-ui-operational-surface + base-ui-and-auth-design)

- **"The base UI is a projection of the system definition, not an application on top of it."**
- Auto-generated from entity definitions. No per-org custom UI. No per-entity custom code.
- **Tables > charts > prose.** Tables are interactive (sort, filter, drill, actions).
- Auto-gen rendering contract: entity list/detail/queue/role-overview views + bootstrap-entity observability + pipeline metric widgets + integration health + runtime observability + cascade viewer + assistant panel.
- Kernel aggregation capabilities: `state-distribution`, `throughput`, `dwell-time`, `queue-depth`, `cascade-lineage`.
- **Assistant at top-bar of every view** (slim, always-visible, `/` or Cmd-K focus). Slide-in panel on engagement.
- **Assistant IS a running chat-harness instance** — not a kernel endpoint. (Current code violates this.)
- Real-time via MongoDB Change Streams filtered by org_id + subscription filters.
- WebSocket keepalive 30-45s ping (Railway proxy requirement).
- `Widget`, `DashboardConfig`, `AlertConfig` deferred post-MVP.
- Active alerting deferred.

## 10. Bulk Operations (bulk-operations-pattern)

- **Pattern, not primitive.** Composes existing mechanisms.
- Generic `bulk_execute` Temporal workflow in kernel code.
- `bulk_operation_id = temporal_workflow_id` (deliberate coupling, documented).
- 5 CLI verbs: `bulk-create`, `bulk-transition`, `bulk-method`, `bulk-update` (silent — bypasses save_tracked), `bulk-delete`.
- Per-batch MongoDB transaction (~1000 ops max per txn).
- Default failure = skip-and-continue; `--failure-mode=abort` opt-in.
- Error classification: permanent (StateMachine/Validation/PermissionDenied/EntityNotFound) skip; transient (VersionConflict/Network/lock-timeout) retry via Temporal.
- Multi-entity per row via skill code (Option α); `bulk_apply` DSL deferred.
- No cross-batch rollback for MVP; saga compensation deferred.
- Terminal states: `completed` / `completed_with_errors` / `failed`.
- CLI monitoring: `indemn bulk status`, `indemn bulk list`, `indemn bulk cancel`, `indemn bulk retry`.

## 11. Actor Model — Spectrum + Skills (primitives-resolved + unified-queue + realtime-architecture)

- **Actor = identity + role + queue + output (CLI commands that change entity state).**
- Three trigger types: message (watch matches), schedule (cron), direct invocation (CLI/API).
- **Actor spectrum**: human / deterministic associate / reasoning associate / hybrid associate. All use same kernel infrastructure.
- **Skill = the program; actor = the executor; interpreter = deterministic or LLM-based.**
- Skills are markdown, stored in MongoDB per-org, content-hashed (tamper detection), version-approved.
- Entity skills: auto-generated from entity definition.
- Associate skills: hand-authored or AI-generated; describe behavior.
- **One queue for humans + associates** (unified-queue insight). Same `findOneAndUpdate` claim. "Associates are employees."
- Rollout is a role assignment change, not infrastructure.
- **`owner_actor_id` on associates** (optional): delegated credential access through owner's Integrations + differentiated auth pattern (session-inherited vs service-token).

## 12. Real-Time + Harness Pattern (realtime-architecture-design)

- **Harness = deployable image outside kernel**, one per kind+framework combination.
- Harness contains: framework (deepagents) + transport (LiveKit/WebSocket/Twilio/Temporal) + harness glue script + `indemn` CLI pre-installed.
- **One harness serves many associates** — loads associate config at session start.
- Harness authenticates to OS via service token (Runtime instance).
- Harness uses CLI subprocess for ALL OS operations. No direct MongoDB. No Python SDK.
- **Mid-conversation event delivery**: scoped watches match Attention → kernel writes `target_actor_id` on message → harness's `indemn events stream` (Change Stream) picks it up → feeds into agent loop.
- **Handoff**: `indemn interaction transfer --to-actor X` or `--to-role Y` → Interaction.handling_actor_id changes → Runtime detects via Change Stream → switches mode.
- **Observation**: Attention.purpose=observing. First-class.
- **Three-layer customer-facing flexibility**: Runtime defaults → Associate override → Deployment override. Merged at session start in harness.
- **Transport bundled with Runtime deployment, NOT a kernel concept.**

## 13. Everything Is Data — Data Architecture

- **OS codebase (Git) = PLATFORM** (deployed once per release).
- **MongoDB = APPLICATION** (per-org config + business data + kernel cross-org data like Platform admin).
- **S3** = unstructured files (scoped by org_id OR actor_id per `content_visibility`).
- **Temporal Cloud** = execution state (no app data).
- **OTEL backend (Grafana Cloud)** = ephemeral traces (days-weeks retention).
- **AWS Secrets Manager** = credentials only (never MongoDB).
- **Separate Atlas clusters** for dev and prod.
- **Clone/diff/deploy** semantics — config only, no instances/messages/secrets.
- **YAML export format**: entities/, roles/, rules/{entity}/, skills/*.md, lookups/, actors/, integrations/, capabilities/.
- **Changes collection = version history + audit trail** (dual purpose).
- **Append-only + hash chain** on changes collection. `indemn audit verify` (still broken per shakeout Finding 14).
- **OrgScopedCollection wrapper** — mandatory org_id injection.
- **PlatformCollection** — cross-org access, platform-admin-only.

## 14. Observability (OTEL baked in)

- `correlation_id = OTEL trace_id`.
- Every primitive touch is a span: entity save, watch eval, rule eval, message creation, Temporal workflow, LLM call, CLI command, HTTP request.
- **3 data stores + 3 purposes + 1 trace_id**:
  - Changes collection: field-level mutations (compliance, years).
  - Message log: completed work items (ops, months-years).
  - OTEL traces: execution path (debugging, days-weeks).
- `indemn trace entity {id}` queries all three, unified timeline.
- `indemn trace cascade {correlation_id}` shows full execution tree.
- Grafana Cloud free tier for MVP.

## 15. Infrastructure + Deployment (infrastructure-and-deployment)

- **Railway for MVP.** US East. Co-located with Atlas + Temporal Cloud.
- **Private networking** (WireGuard-encrypted). Internal services (QP, TW, harnesses) not public.
- **Shared variables** for cross-service config + AWS creds.
- **WebSocket keepalive** mandatory (30-45s ping).
- **External health monitoring**: Grafana Cloud HTTP check on API + MongoDB heartbeat records for QP/TW.
- **MongoDB connection pool sizing**: API 50/replica, QP 10, TW 30/replica.
- **Cloudflare before first external customer** (WAF + DDoS).
- **Optimistic dispatch + sweep backstop**.
- **Scaling is manual on Railway** (replica count slider).
- **Cost**: ~$200/mo MVP → ~$250/mo first customer → ~$420/mo 5 customers → ~$1,700/mo 50 customers.
- **Kernel image** = `indemn-kernel` with 3 entry points (api, queue_processor, temporal_worker).
- **Harness images** = `indemn/runtime-{kind}-deepagents` (3 kind+framework images).
- **Dependencies**: MongoDB Atlas is only SPOF. Temporal down → associates pause, humans continue. LLM down → deterministic ops continue via `--auto`. Secrets Manager down → cached credentials serve. Multi-provider LLM resilience (per-associate provider config).
