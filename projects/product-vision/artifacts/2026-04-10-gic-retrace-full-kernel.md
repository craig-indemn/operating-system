# GIC Retrace Against the Updated Kernel

## Purpose

Session 4 substantially evolved the architecture: Integration elevated to primitive #6, Temporal for associate execution, unified MongoDB queue, everything-is-data, watches as wiring, the `--auto` pattern for kernel capabilities. This retrace validates the full GIC email intelligence pipeline against that updated kernel. The original GIC trace was done before these changes and needs to be refreshed.

The question this artifact answers: **does the kernel hold up end-to-end when you walk through a real production system?** If yes, we're close to spec-ready. If no, the gaps need to be named and resolved.

## The Updated Primitive Set (Six)

1. **Entity** — domain data with structure, lifecycle, CLI/API, auto-generated skills
2. **Message** — work items generated by entity changes, routed by watches
3. **Actor** — identity (human or automated) that processes triggers via CLI
4. **Role** — permissions + watches
5. **Organization** — multi-tenancy scope
6. **Integration** — external system connection (provider, credentials, owner, adapter dispatch)

## GIC Setup on the Kernel

### Domain entities

All created as data in MongoDB via `indemn entity create`:

- **Email** — inbound messages from Outlook. Fields: `from_address`, `to_addresses`, `subject`, `body`, `has_attachments`, `attachments` (S3 refs), `classification`, `classification_type`, `lob`, `submission_id`, `thread_id`, `received_at`
- **Extraction** — structured data extracted from email attachments. Fields: `email_id`, `source_file`, `extracted_data` (flexible), `confidence`
- **Submission** — the work unit for underwriting. Fields: `named_insured`, `lob`, `stage`, `carrier_id`, `agent_id`, `external_ref`, `effective_date`, `data` (flexible per product), `last_activity_at`, `followup_count`, `is_stale`, `ball_holder` (computed)
- **Assessment** — AI analysis of submission state. Fields: `submission_id`, `situation_type`, `completeness_score`, `missing_fields`, `next_action`, `draft_appropriate`, `needs_review`
- **Draft** — generated response drafts. Fields: `assessment_id`, `recipient`, `subject`, `body`, `draft_type`, `requires_approval`, `status`
- **Carrier** — carrier reference. Fields: `name`, `contact_info`, `binding_authority`, `lob_appetite`
- **Agent** — retail agent reference. Fields: `name`, `email`, `agency`, `producer_code`

## The End-to-End Flow

### Step 1: Email sync (scheduled → Integration → entity creation)

**Trigger**: Cron fires every 1 minute. The kernel scheduler creates a queue item targeting role `email_sync`.

**Execution**:
1. Queue Processor reads the queue item. Role `email_sync` has an associate attached, so it starts a Temporal workflow.
2. Temporal workflow claims the message from the queue (`findOneAndUpdate`).
3. Workflow loads the `Email Sync` associate's skill (one-line: "run `indemn email fetch-new`").
4. Workflow executes the command (a method on the Email entity type).
5. The `external-sync` method on Email resolves the Integration: queries for `system_type=email, owner_type=org, org_id=gic`. Finds INT-OUT-001 (Outlook v2).
6. Method reads `provider_version` → `outlook_v2`. Fetches adapter from registry: `OutlookV2Adapter`.
7. Method reads `secret_ref` → fetches credentials from AWS Secrets Manager at `/indemn/prod/org/gic/integration/INT-OUT-001`.
8. Instantiates adapter with credentials. Calls `adapter.fetch(since=last_sync_timestamp)`.
9. Adapter hits Graph API, returns 10 new messages with metadata.
10. For each new message:
    - Adapter downloads attachments to S3 at `s3://indemn-files/gic/email/{message_id}/{attachment_name}`
    - Constructs Email entity data (maps Outlook format to OS Email fields)
11. **Bulk transaction**: all 10 Email entities created + 10 message_queue events written in one MongoDB transaction.
12. Transaction commits. 10 Email:created events now visible to watches.
13. Workflow marks the original email_sync queue item complete.

**What the system sees after**: 10 new Emails in the database, 10 Email:created messages in the queue waiting for watches to match.

### Step 2: Extraction (for emails with attachments)

**Trigger**: `extractor` watch matches `Email:created[has_attachments=true]`.

**Execution**:
1. For each matching Email, Queue Processor starts a Temporal workflow for the extractor associate.
2. Workflow claims the message, loads the Email entity.
3. Loads extractor's skill: "Extract structured data from each attachment. For each, determine the document type (ACORD, loss run, supplemental app, etc.) and extract the fields relevant to that type."
4. For each attachment (S3 ref on the Email):
   - Downloads the file from S3
   - Calls `indemn email pdf-extract --attachment {name}` (a method on Email, activated via the `pdf-extract` kernel capability)
   - The capability does OCR + LLM extraction
   - Returns structured data
5. **Multi-entity transaction**: creates N Extraction entities (one per attachment) + updates Email.extraction_status.
6. Transaction commits. N Extraction:created events + 1 Email:fields_changed event in the queue.
7. Workflow marks complete.

### Step 3: Classification (hybrid — rules first, LLM fallback)

**Trigger**: `classifier` watch matches. Two entry paths:
- `Email:created[has_attachments=false]` — immediate classification (no extraction needed)
- `Extraction:created` — classify after extraction completes

**Execution** (same either way):
1. Classifier workflow claims the message, loads the Email (and linked Extractions if any).
2. Skill instructs: "Classify this email. Call `indemn email classify {email.id} --auto`. If the response includes `needs_reasoning: true`, analyze the email content and extraction data, determine the classification type and LOB, then call classify with explicit values."
3. Executes `indemn email classify EMAIL-001 --auto`:
   - The `auto-classify` method loads rules for Email in this org (cached, 60s TTL)
   - Evaluates rules in priority order, including vetoes first
   - For an email from `agent@usli.com` with subject `MGL123ABC Quote Request`:
     - Veto rule checks: does subject contain "Decline"? No. Veto does not fire.
     - USLI rule matches: `from_address ends_with @usli.com` → sets `classification: "usli_quote"`
   - Extracts `MGL` prefix from subject, looks up in `usli-prefix-lob` → `general_liability`
   - Sets `lob: "general_liability"`
   - Returns success (no needs_reasoning)
4. The method modifies Email.classification, Email.classification_type, Email.lob and saves. **Selective emission**: the save produces one event with metadata `{method: "classify", fields_changed: ["classification", "classification_type", "lob"]}`.
5. Transaction commits. One Email:method_invoked event in the queue.
6. Workflow marks complete.

**If needs_reasoning had come back** (e.g., unknown sender domain): the skill would instruct the LLM to read the email content, extraction data, and attached documents, determine the type and LOB, then call classify explicitly. The LLM is only invoked when the deterministic rules can't decide.

### Step 4: Linking (hybrid)

**Trigger**: `linker` watch matches `Email:method_invoked[method=classify]`.

**Execution**:
1. Linker workflow claims the message, loads the Email.
2. Skill instructs: "Link this email to an existing submission or create a new one. Call `indemn email link {email.id} --auto`. If needs_reasoning, you'll receive candidates — pick the best match or create new."
3. Executes `indemn email link EMAIL-001 --auto`:
   - The `auto-link` method tries reference patterns first
   - For GIC: patterns include USLI refs (`[A-Z]{2,3}\\d{3}[A-Z0-9]{3,}`) and GIC numbers (`\\d{6}`)
   - Extracts candidate refs from subject and body: finds `MGL123ABC`
   - Searches Submission where `external_ref = "MGL123ABC"` (scoped to gic org)
   - Found one match → succeeds, links Email to Submission
4. The method updates Email.submission_id and saves. One Email:fields_changed event emitted.

**If the reference search failed**: auto-link would fall back to fuzzy search on `named_insured` (extracted from email content earlier or requested from the LLM). If single good match → links. If ambiguous → returns needs_reasoning with candidate Submissions attached.

**If no match at all**: needs_reasoning with "no candidates found, consider creating new." The skill would have the LLM decide: create new or leave unlinked. For create-new, it calls `indemn submission create` with extracted details.

**Multi-entity case**: When creating a new Submission, the linker's transaction includes: Submission create + Email update (setting submission_id) + Agent lookup/create if needed. All in one MongoDB transaction. Crash recovery via Temporal replay.

### Step 5: Assessment (mostly reasoning)

**Trigger**: `assessor` watch matches `Submission:created`.

**Execution**:
1. Assessor workflow claims, loads Submission + linked Emails + Extractions.
2. Skill instructs: "Assess the submission. First call `indemn submission completeness-check {submission.id}` to determine what's present and what's missing per the LOB's required fields. Then analyze the full context — email threads, extraction data, situation history — to determine the situation type (new submission, follow-up, quote received, decline, etc.) and the appropriate next action."
3. Executes `indemn submission completeness-check SUB-001`:
   - The `completeness-check` method reads required-field-rules for this org + LOB
   - Checks which required fields are present in Submission.data
   - Returns `{score: 0.7, missing_fields: ["business_description", "revenue"]}`
4. LLM reads the full context and produces an assessment object.
5. **Multi-entity transaction**: creates Assessment entity (with situation_type, completeness_score, next_action, draft_appropriate, needs_review) + updates Submission.stage=triaging.
6. Transaction commits. Two events: Assessment:created + Submission:stage_changed.
7. Workflow marks complete.

### Step 6: Draft generation (reasoning)

**Trigger**: `draft_writer` watch matches `Assessment:created[draft_appropriate=true]`.

**Execution**:
1. Draft writer claims, loads Assessment + Submission + linked Emails.
2. Skill instructs: "Generate an appropriate response draft based on the assessment's situation_type and next_action. For 'missing_info', draft an info request listing the specific missing fields. For 'acknowledgment', draft a receipt confirmation. For 'quote_communication', draft a quote presentation."
3. LLM generates the draft text.
4. **Transaction**: creates Draft entity with status=pending_review, requires_approval (true if high-stakes, false for routine acknowledgments).
5. One Draft:created event emitted.
6. Workflow marks complete.

### Step 7: Human review (underwriter)

**Trigger**: `underwriter` watch matches multiple events. In this case, `Assessment:created[needs_review=true]` for a low-confidence assessment, or `Draft:created[requires_approval=true]` for a draft that needs human sign-off.

**Execution** (human path):
1. Queue Processor writes the message to message_queue targeting role `underwriter`. **Underwriter role has no associate attached** (or has an optional associate that JC has trained to handle routine cases).
2. The message appears in JC's UI queue (rendered from a query on message_queue).
3. JC reviews the draft. Clicks "Approve." UI calls `indemn draft transition DRAFT-001 --to approved`.
4. The transition method changes Draft.status to approved, saves. One Draft:method_invoked event emitted (or Draft:transitioned[to=approved]).
5. UI marks the message complete in the queue.

### Step 8: Outbound send (deterministic associate, Integration-backed)

**Trigger**: `operations` role watches `Draft:transitioned[to=approved]`. An `Outbound Sender` associate has the operations role.

**Execution**:
1. Outbound sender claims, loads Draft.
2. Skill instructs: "Send this approved draft as an email via the configured email integration. Call `indemn email send --from-draft {draft.id}`."
3. Executes `indemn email send`:
   - The `send` method on Email type resolves the org's email Integration (same one email_sync uses)
   - Adapter sends via Graph API
   - Returns success + external message ID
4. Updates Draft.status=sent, stores external_ref.
5. Workflow marks complete.

### Step 9: Stale detection (scheduled, fully deterministic)

**Trigger**: Cron fires every 1 hour. Schedule creates queue item for `stale_checker` role.

**Execution**:
1. Stale checker workflow claims, runs skill: "Run `indemn submission check-stale`."
2. Executes the method:
   - The `stale-check` method evaluates the configured condition `last_activity_at older_than 7d AND followup_count >= 2`
   - Finds matching Submissions (could be 0, 5, or 50)
   - Bulk-updates `is_stale=true` on each
   - **Single transaction**: N Submission updates + N message_queue events emitted atomically
3. Transaction commits. N Submission:fields_changed[is_stale=true] events in the queue.
4. Workflow marks complete.

### Step 10: Operations triage

**Trigger**: `operations` watch matches `Submission:fields_changed[is_stale=true]` (plus other ops watches).

**Execution**: Each stale submission generates a message in Maribel's queue. Maribel reviews, takes action (followup, escalate, close). Each action changes the submission stage, which propagates through watches.

## What Held Up

1. **Integration as primitive works cleanly.** Email sync resolves its Outlook connection through the Integration entity. Credentials flow through Secrets Manager. Adapter versioning is per-integration. No special-case code paths for "integrations vs. entities."

2. **Watches handle all orchestration.** Zero `processing_status` fields on Email to encode pipeline ordering. Each step watches the previous step's emitted event. The pipeline emerges from watches, not from explicit workflow orchestration.

3. **Multi-entity atomicity works for single-turn operations.** Linker's "create Submission + update Email," assessor's "create Assessment + update Submission stage," and draft writer's "create Draft + update Assessment" all fit in single MongoDB transactions. No sagas needed at this level.

4. **The `--auto` pattern eliminates LLM waste.** Classification, linking, and completeness-check all attempt deterministic rules/patterns first. LLM is only invoked on the cases that deterministic logic can't handle (unknown senders, ambiguous fuzzy matches, complex assessments). Cost scales with edge case complexity, not volume.

5. **Unified queue works for humans and associates.** JC and Maribel claim from the same MongoDB queue as the associates. A role transition (removing JC from classifier when an associate is trusted) requires no infrastructure change.

6. **Correlation ID chains through the full pipeline.** From Email:created at step 1 to Draft:sent at step 8, 8+ hops across multiple actors and transactions, all tagged with the same correlation_id. `indemn trace cascade CORRELATION-ID` produces the full execution tree.

7. **Temporal wraps associate execution with durability.** Each associate's processing is a Temporal workflow activity. Crash during linker processing → replay picks up from the last completed step. No orphaned entities, no stuck messages (visibility timeout + replay handle both).

8. **Scheduled work uses the same path as message-triggered work.** Stale check, email sync, and future scheduled tasks all create queue items that flow through the same associate claiming mechanism. One model for all actor activation.

9. **The kernel-vs-domain split is clean.** GIC's business logic (classification rules, reference patterns, stale thresholds, completeness requirements) is entirely in per-org configuration (rules, lookups, required-field-rules, capability activations). The kernel has zero insurance assumptions. A completely different domain (content management, sales CRM) would use the same kernel with different data.

## What Surfaced

### 1. Bulk event emission creates human-queue spikes

When the stale checker flags 50 submissions in one transaction, 50 messages are emitted to the operations queue. Maribel opens her queue and sees 50 rows appear at once with no context.

**Classification**: UX concern, not an architectural break. The kernel correctly emits 50 events. The problem is how the UI (and human actor) consume them.

**Proposed mechanism**: watch-level coalescing. A watch can declare a coalescing strategy: "batch messages from the same source event (same parent correlation_id) within a time window into one queue item." The batch has a summary and drill-down. The kernel supports this by tagging emitted messages with a batch_id when appropriate, and the queue query joins on batch_id.

```bash
indemn role add-watch operations \
  --entity Submission \
  --event fields_changed \
  --when '{"field": "is_stale", "op": "equals", "value": true}' \
  --coalesce '{"strategy": "by_correlation_id", "window": "5m", "summary": "{{count}} submissions flagged stale"}'
```

This is a new kernel feature that didn't exist in the previous design. It's additive — if no coalescing is configured, behavior is unchanged (one message per event).

### 2. Pipeline overview is not queue-shaped

A user asking "how many submissions are in each stage? how many emails are awaiting classification? how fast are we processing?" is not asking for a queue — they're asking for a dashboard. Watches give you work-to-do; entity queries give you state distribution.

**Classification**: Missing capability in the current architecture. Not a break, but a gap.

**Proposed**: Auto-generated dashboards are part of the base UI, sourced from:
- State machine distribution per entity type (pie chart of submissions by stage)
- Throughput metrics from the message log (emails processed per hour, average time from received to quoted)
- Queue depth per role (from message_queue)
- Error/retry rates from Temporal workflow history
- Integration health (status of each active Integration)

These are built from the existing primitives (entities, message log, Temporal history, Integration). No new primitives needed. Just a rendering layer that aggregates and presents.

### 3. Human-in-progress review soft-lock

JC opens a draft to review. Walks to a meeting. Another user grabs it. Race condition on review.

**Classification**: UI + kernel coordination concern. The kernel's existing `claimed` state on messages is for explicit claiming after deciding to act. "Soft-lock while viewing" is a different state.

**Proposed mechanism**: ephemeral locks on entities, not messages. When a UI opens an entity for interactive review, it calls `indemn entity lock ENTITY-ID --as me --duration 5m --purpose review`. The lock is informational: other UIs see "JC is reviewing this" and can proceed with caution. Heartbeat keeps it alive during active review. Auto-expires on inactivity.

This is a small addition — one CLI command, one collection (or fields on the entity document) to track locks. Works for any entity, not just drafts.

### 4. Multi-LOB draft consolidation

One email is a request for GL + Umbrella → linker creates two Submissions in one transaction → two assessments → two drafts → the agent gets two emails instead of one.

**Classification**: Domain logic concern, not kernel.

**Resolution**: The draft_writer skill can query for other Drafts with the same source_email_id or same parent correlation_id and coalesce them into one outbound message. This is a skill-level decision, not a kernel mechanism. Flagged so the skill author knows.

### 5. Queue health and pipeline health need first-class tools

When something stops working — an associate crashes, an Integration goes into error state, a bottleneck forms at a specific role — how does ops (or the platform team) notice before customers complain?

**Classification**: Observability tooling concern. The data exists; the tools to surface it don't yet.

**Proposed CLI** (should be part of the spec):
- `indemn queue show --role underwriter` — current queue depth and oldest item
- `indemn queue stats` — depth per role across the org
- `indemn associate health` — last run, failure rate, average duration per associate
- `indemn integration health` — status, last check, recent errors per Integration
- `indemn trace cascade CORRELATION-ID` — full execution tree
- `indemn metrics throughput --entity Email --period 1d` — processed rate per entity type

All of these are reads against existing stores (message_queue, message_log, changes, Integration status, Temporal workflow history).

### 6. Ball holder as computed field — activation config scope

The stage → ball_holder mapping is intrinsic to the Submission entity in GIC's setup. It's not a reusable lookup; it's the specific mapping for this entity's computed field.

**Resolution**: The mapping belongs on the method activation, not as a separate Lookup. When calling `indemn entity add-method Submission computed-field --name ball_holder --from stage --mapping {...}`, the mapping is stored with the activation config, not as a shared org-level Lookup. Lookups are for data that rules reference at runtime (where bulk import / shared editing matters). Method activation configs are for intrinsic entity behavior.

This is a clarification to document, not a change to the architecture.

### 7. The scheduled-task-as-synthetic-queue-item pattern

When the cron fires for stale_check, it creates a queue item targeting role `stale_checker`. The stale_checker associate claims it and runs. This works, but the queue item is purely a synthetic activation signal — it carries no entity reference, no "what happened" metadata.

**Classification**: Minor aesthetic concern. The pattern is consistent (everything flows through the queue), but the synthetic messages are a slightly different shape than real event-driven messages.

**Assessment**: This is fine. The benefit (one activation model for all actors) outweighs the cost (one variant of message schema). The synthetic message just has `event_type: "schedule_fired"` and references the schedule definition. Queue operations work uniformly.

## Validation Verdict

**The architecture holds up for GIC.** Every step of the production pipeline maps cleanly to the 6 primitives. No gaps that break the kernel. No primitives that feel forced or superfluous. The `--auto` pattern makes LLM cost proportional to complexity. Multi-entity atomicity is handled by MongoDB transactions, not sagas. The unified queue works for mixed human-associate roles. Integration as a primitive formalizes what the architecture already depended on.

**Six concerns surfaced, none architectural:**
1. Bulk event coalescing for human queues (new kernel feature, additive)
2. Pipeline dashboards as base UI feature (auto-generated, no new primitives)
3. Ephemeral entity locks for UI soft-lock (small kernel addition)
4. Multi-LOB draft consolidation (domain skill concern, not kernel)
5. Queue and integration health tooling (CLI commands over existing data)
6. Computed field mapping config scope (clarification)

None of these block GIC running on the OS. All are additions or clarifications. The kernel-vs-domain split holds — GIC's business logic is entirely in per-org data, the kernel has zero insurance assumptions.

**Next retraces**: EventGuard (end-to-end quoting/binding/payment, pressure-tests distribution surfaces and real-time channels) and a CRM use case (pressure-tests that the kernel isn't secretly insurance-shaped). If those also hold up cleanly, the architecture is ready to encapsulate as a white paper and start building from.
