**Learning**: `voice-service/scripts/` dir doesn't exist — `.env.aws` is generated by a GitHub Actions runner on deploy. But Parameter Store drift between dev and prod was the actual problem, not code. Check `aws ssm get-parameter` before writing defensive code against apparent API quirks.

## Patterns worth carrying forward

1. **The per-repo env-loader pattern** (`scripts/aws-env-loader.sh`) is what operations_api, voice-service, voice-livekit, middleware-socket-service, and others use. Any new service should adopt it; any new env var on an existing service should go to PS/SM, not hand-edited `.env.aws`. See devops project INDEX for references.

2. **Hand-maintained `.env.aws` is a trap** when services also run aws-env-loader. Hand edits get overwritten on next deploy. Always check whether a service regenerates its env file before touching `.env.aws`.

3. **"Just use what they gave" caveat** — when a teammate DMs a block of env vars, don't blindly overwrite existing AWS values. Cross-check: does the existing code consume the var in multiple places? Will changing it break unrelated flows? Pre-flight those questions, or explicitly confirm.

4. **The disk-full pattern** on dev EC2s (`dev-services`, `voice-livekit`) is recurring. Future: auto-prune or size-up. For now the recipe is the same every time: `df -h` → `docker system df` → `docker image prune -a -f` → rerun failed deploy.

5. **The broken-repo-from-external-fork pattern** (form-extractor) may recur if more repos get imported from personal forks. Checklist: GitHub Actions secrets, self-hosted runner registration, deploy script's `cd /opt/<dir>` target, compose file's image source.

6. **Cross-workspace file downloads don't work with Indemn-only Slack token** — UG workspace files can't be pulled via API. Either user pastes or downloads manually, or we'd need to import UG workspace credentials to `agent-slack`.
