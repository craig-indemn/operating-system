## Key Claims

7 architectural issues resolved (Round 1):
- **#7 Skill vs. workflow orchestration (HIGH RISK)**: Resolved — Temporal workflow is GENERIC (claim → process → complete). Skill is source of truth for orchestration. No per-actor workflow code.
- **#3 CLI entity creation vs. code (HIGH RISK)**: Resolved — CLI generates Python class files; hand-written and CLI-generated are identical. One entity model. (LATER EVOLVED: entity definitions live in MongoDB, not Python files — per data-architecture-everything-is-data.)
- **#2 Outbox vs. message_queue (MEDIUM)**: Resolved — eliminate separate outbox. Write directly to message_queue in entity save transaction.
- **#1 Scheduled vs. message-triggered actors (MEDIUM)**: Resolved — schedules create queue items, same path. All work through queue.
- **#5 Real-time direct invocation vs. queue (MEDIUM)**: Resolved — queue entry + direct invocation simultaneously. Direct invocation claims queue entry immediately. Work is visible in queue; execution bypasses dispatch for latency.
- **#4 Organization as primitive vs. entity (LOW)**: Resolved — Organization IS an entity the kernel understands specially. Same for Actor, Role. "Bootstrap entities."
- **#6 Context enrichment depth (LOW)**: Resolved — configurable per watch, default shallow. Actors fetch more via CLI if needed.

## Architectural Decisions

- **Generic single workflow** for associate processing (no per-associate workflow code).
- **Outbox eliminated** — entity save + watches + messages = one MongoDB transaction.
- **Schedules → queue items** (synthetic messages with `event_type: scheduled_task`).
- **Real-time = queue + direct invocation in parallel** (direct invocation claims the queue entry).
- **Bootstrap entities**: Organization, Actor, Role are entities that the kernel knows specially. Later: 7 kernel entities (add Integration, Attention, Runtime, Session).
- **Entity creation via CLI generates Python class file** — later superseded by "entity definitions live in MongoDB + dynamic class creation at startup" (2026-04-09-data-architecture-everything-is-data.md).

## Layer/Location Specified

- **Temporal workflow** = generic `ProcessMessageWorkflow` (3 activities). Location: left open here (resolved as kernel Temporal Worker by Phase 2-3 spec; per realtime-architecture-design should be harness).
- **Entity definitions** = Python class files (this artifact) → MongoDB data (later). Code in `kernel/entity/factory.py` creates dynamic classes.
- **Message queue** = MongoDB, kernel-managed, single collection.
- **Scheduled tasks** = kernel scheduler writes queue items; actor claims same as event-triggered.
- **Real-time direct invocation** = API endpoint in kernel (`kernel/api/direct_invoke.py` per Phase 2-3).
